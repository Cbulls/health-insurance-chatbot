# harag — 성능 측정 설계 (Performance Measurement Plan)

이 문서는 harag의 성능을 **재현 가능하게 측정하고 SLO 부합을 판정**하는 방법을 정의한다.
측정 도구는 이미 코드에 있다(`eval/perf.py`, `eval/capacity.py`, `observability/tracing.py`).
이 문서는 그 위에 "무엇을·어떻게·어떤 기준으로" 측정할지를 설계한다.

핵심 원칙 — **측정은 숫자 뽑기가 아니라, 어디서 무엇이 느린지와 SLO를 지키는지를 판정하는 일.**

---

## 1. 측정 철학 (먼저 합의할 것)

성능 측정에서 흔한 거짓말 네 가지를 우리는 코드로 방어한다:

| 함정 | 방어 |
|---|---|
| **평균의 거짓말** — 평균은 꼬리 지연을 숨긴다 | p50/p95/p99 백분위로 본다(`perf.py`) |
| **워밍업 오염** — 첫 요청들(콜드 캐시·JIT)이 통계를 왜곡 | 앞 N개 제외(`analyze(warmup=N)`) |
| **단일 숫자의 함정** — "느리다"는 알아도 "어디가"를 모름 | 단계별 분해(파싱·임베딩·검색·리랭킹·생성) |
| **closed-loop 거짓말** — 부하기가 응답을 기다리면 실제 부하를 못 만듦 | open-loop 부하(도착률 고정) |

**측정 전에 SLO를 동결한다(굿하트 방어).** 측정 후 "이 정도면 됐다"고 기준을 맞추면 의미가 없다.

---

## 2. 무엇을 측정하는가 — 두 경로

### 2.1 읽기 경로 (질의 → 응답) — 사용자 체감 지연
사용자가 질문하고 답을 받기까지. **end-to-end가 주 지표, 단계별이 진단 지표.**

```
질문 → [재작성] → [검색] → [리랭킹] → [생성(LLM)] → 응답
        rewrite    retrieval  rerank     generation
```

측정 항목:
- **E2E 총 지연** (질문 수신 → 응답 완료)
- **TTFT** (Time To First Token) — 스트리밍이면 첫 토큰까지. 체감 반응성의 핵심.
- **단계별 지연** — 각 단계가 전체에서 차지하는 비중(병목 식별)
- **abstention 경로 지연** — "모른다" 응답은 LLM을 안 부르므로 빠름. 별도 집계.

### 2.2 쓰기 경로 (문서 → 검색 가능) — 인덱싱 처리량
문서를 올리고 검색 가능해지기까지. **사용자 대기가 아니라 처리량·격리가 관심.**

```
업로드 → [파싱] → [PII] → [청킹] → [임베딩] → [적재] → [버전전환] → 검색가능
         parse           chunk     embed       upsert    activate
```

측정 항목:
- **문서당 인덱싱 시간** (파싱→검색가능, 단계별 분해)
- **처리량** (docs/분, chunks/초)
- **증분 효과** — 일부 변경 시 전체 대비 단축률
- **읽기 격리 확인** — 인덱싱 부하 중 질의 지연이 악화되지 않는가(NFR-2)

## 3. 단계별 측정 방법

### 3.1 계측 지점 (이미 있는 도구 활용)
`observability/tracing.py`의 `QueryTrace.stage()` 컨텍스트 매니저로 각 단계를 감싼다:

```python
with trace.stage("retrieval"):
    results = retriever.retrieve(...)
with trace.stage("rerank"):
    results = reranker.rerank(...)
with trace.stage("generation"):
    answer = generator.generate(...)
```

수집된 `LatencySample(total, stages={...})`를 `perf.analyze()`로 백분위 분석.

### 3.2 쓰기 경로 단계 계측
워커에도 동일하게 단계 타이머를 두른다: parse / pii / chunk / embed / upsert / activate.
**임베딩이 보통 최대 병목**(GPU 추론) — 단계 분해로 확인하고 배치 크기·동시성 튜닝의 근거로.

### 3.3 측정 산출물 예시
```
읽기 경로 E2E: n=1000 p50=420ms p95=1180ms p99=2300ms
  단계별 p95: rewrite=15ms, retrieval=85ms, rerank=140ms, generation=920ms
  → 병목: generation(LLM). 리랭킹·검색은 여유.
abstention 경로: p95=180ms (LLM 미호출로 빠름)
```

## 4. 부하 테스트 설계

### 4.1 부하 모델 — open-loop (도착률 고정)
closed-loop(요청 완료 후 다음 요청)는 시스템이 느려지면 부하도 자동으로 줄어 **실제 피크를 재현 못 한다.**
대신 **도착률(RPS)을 고정**하고 주입한다. 도구: k6 또는 locust(`--constant-arrival-rate`).

### 4.2 부하 단계 (ramp)
| 단계 | 목적 |
|---|---|
| **베이스라인** (1 RPS) | 무부하 기준 지연 |
| **목표 부하** (예상 피크 RPS) | SLO 부합 확인 |
| **스트레스** (목표의 2~3배) | 한계점·degradation 동작 확인 |
| **스파이크** (급증) | 오토스케일·큐잉 반응 |
| **소크** (목표 부하 1시간+) | 메모리 누수·성능 열화 |

### 4.3 워크로드 구성 (현실 반영)
- 질의 유형 혼합: 일반 질의 70% / abstention 유발 15% / 멀티턴 15%
- 권한 분포: 좁은 ACL ~ 넓은 ACL 혼합(필터 비용 차이 측정)
- 캐시 무효화: 동일 질의 반복은 캐시를 타므로, 질의 다양성 확보

### 4.4 읽기/쓰기 동시 부하 (NFR-2 검증)
질의 부하를 거는 **동시에** 대량 인덱싱을 주입 → 질의 p95가 악화되지 않는지 확인.
악화되면 읽기/쓰기 격리가 깨진 것(큐·워커 분리 재점검).

---

## 5. 용량 산정 (`eval/capacity.py`)

부하 곡선에서 **"몇 RPS까지 SLO를 유지하는가"**를 용량으로 확정한다.
- RPS를 올리며 p95를 기록 → SLO를 넘기 시작하는 지점이 단일 인스턴스 용량.
- 역산: 목표 동시 사용자 → 필요 RPS → 필요 인스턴스/GPU 수.
- 임베딩 GPU는 별도 산정(인덱싱 배치 + 질의 임베딩 부하).

## 6. 기능 요구사항(FR) 부합 측정

성능과 별개로, **기능이 정확히 동작하는지**를 정량 측정한다. (품질 게이트, `eval/harness.py`)

| FR | 측정 지표 | 판정 기준 |
|---|---|---|
| FR-2 파싱 | 표 복원율(층화) | PoC 임계 이상 (`eval/poc.py`) |
| FR-3 청킹 | 조항 보존율, 폴백 발동률 | 폴백률 임계 이하 |
| FR-5 검색 | recall@k, MRR | SLO 신뢰구간 하한 초과 |
| FR-6 멀티턴 | 재작성 후 검색 성공률 | 지시어 해소율 |
| FR-7 리랭킹 | 리랭킹 전후 정답 순위 상승 | nDCG 개선 |
| FR-8 생성 | faithfulness(근거 충실도), abstention 정확도 | 신뢰구간 하한 초과 |
| FR-9 인용 | 인용 정확도(위조 0건) | 위조 인용 차단율 100% |

**품질 게이트 핵심:** 단순 평균이 아니라 **부트스트랩 신뢰구간 하한이 SLO를 넘어야 통과**
(`eval/quality.py`). 작은 골드셋의 우연한 통과를 막는다.

---

## 7. 비기능 요구사항(NFR) 부합 측정

| NFR | 측정 방법 | 판정 기준 |
|---|---|---|
| **NFR-2 읽기/쓰기 격리** | 인덱싱 동시 부하 중 질의 p95 비교 | 질의 p95 악화 ≤ X% |
| **성능(지연)** | open-loop 부하에서 E2E p95/p99 | p95 ≤ SLO, p99 ≤ SLO×K |
| **처리량** | 인덱싱 docs/분, 질의 RPS | 목표 이상 |
| **NFR-4 장애 대응** | LLM 타임아웃·rate limit 주입 시 동작 | degradation 작동, 에러율 임계 이하 |
| **NFR-5 보안** | ACL 누수 검사(부하 중에도) | 권한 외 회수 0건 |
| **가용성** | 소크 테스트 중 에러율·지연 추이 | 열화 없음 |
| **확장성** | 인스턴스 추가 시 처리량 선형성 | 스케일 효율 ≥ 임계 |

## 8. 측정 환경·절차

### 8.1 환경 분리
- **측정은 운영과 동일 사양 스테이징에서.** 개발 노트북 측정은 무의미.
- 실제 모델(임베딩/리랭커 GPU, LLM API), 실제 저장소(Qdrant·PG·오브젝트), 실데이터에 준하는 코퍼스.
- 측정 환경·커밋 해시·모델 버전·데이터셋 버전을 **결과에 함께 기록**(재현성).

### 8.2 절차
1. SLO 동결(측정 전) → `settings.py`의 플레이스홀더를 목표값으로
2. 워밍업 실행(통계 제외분)
3. 부하 단계별 실행, 단계별 trace 수집
4. `perf.analyze(warmup=N)`로 백분위·단계 분해
5. SLO 판정(`LatencyReport.passes(slo)`), 용량 산정
6. 결과 리포트 + 병목·개선 권고

### 8.3 CI 연계
- `quality-gates.yml`(야간)에 성능 회귀 검사 추가: 베이스라인 대비 p95 악화 시 알림.
- PR 차단은 아님(부하 테스트는 무겁고 환경 필요) — 야간 추세 모니터링.

---

## 9. 대표 측정 시나리오 (체크리스트)

서비스 출시 전 반드시 측정·판정할 항목:

- [ ] **단일 질의 E2E 지연** (무부하) — 단계별 분해, 병목 식별
- [ ] **목표 부하 SLO 부합** — p95/p99가 동결 SLO 이내
- [ ] **스트레스 한계점** — 어느 RPS에서 무너지는가 + degradation 동작
- [ ] **읽기/쓰기 격리** — 인덱싱 부하 중 질의 지연 악화율 (NFR-2)
- [ ] **문서 인덱싱 시간** — 파싱→검색가능, 문서 크기별
- [ ] **증분 인덱싱 효과** — 일부 변경 시 단축률
- [ ] **abstention 경로 지연** — LLM 미호출 경로가 빠른가
- [ ] **장애 주입** — LLM 타임아웃·rate limit 시 동작 (NFR-4)
- [ ] **부하 중 ACL 무결성** — 권한 외 회수 0건 (NFR-5)
- [ ] **소크 테스트** — 장시간 열화 없음
- [ ] **FR 품질 게이트** — recall/faithfulness/인용 정확도 신뢰구간 통과
- [ ] **용량 산정** — 목표 동시 사용자 → 필요 인스턴스/GPU

---

## 10. 한계 — 측정으로도 못 잡는 것 (정직하게)

- **실사용 분포와 부하 워크로드의 괴리** — 합성 부하는 실제 질의 분포를 완벽히 못 흉내. 출시 후 실측 모니터링으로 보정.
- **LLM API 외부 변동성** — 외부 프로바이더의 지연은 우리 통제 밖. p99 꼬리에 영향. 측정에 포함하되 변동을 인지.
- **품질 SLO의 골드셋 의존** — 측정 정확도는 골드셋 대표성에 달림(별도 라벨링 투자).
- **콜드스타트** — 오토스케일 신규 인스턴스의 콜드 캐시·모델 로드 지연은 워밍업과 별개로 관리.

---

## 요약

측정은 세 질문에 답해야 한다:
1. **얼마나 빠른가?** — E2E p95/p99 + 단계별 병목 (읽기·쓰기 경로)
2. **SLO를 지키는가?** — 동결된 기준 대비 부하 단계별 판정
3. **기능이 정확한가?** — FR 품질 게이트(신뢰구간), NFR 부합(격리·보안·장애)

도구는 이미 있다(`perf.py`·`capacity.py`·`tracing.py`·`harness.py`·`quality.py`).
남은 건 **실제 스테이징 환경에서 실데이터로 돌리고, SLO를 동결해 판정**하는 일이다.
