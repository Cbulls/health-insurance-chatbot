# harag — 서비스화 로드맵 (실제 운영까지 남은 것)

지금까지는 "코드가 올바른가"였다. 이 문서는 "실제 서비스로 뜨는가"다 — 다른 문제다.
2026년 RAG 프로덕션 모범 사례 조사를 기반으로, 현재 상태와 남은 작업을 정리한다.

조사 출처: 2026년 RAG 프로덕션 체크리스트·거버넌스 가이드·K8s LLM 서빙 가이드·HWP 파싱 라이브러리 현황.

---

## 0. 한눈에 — 우리가 이미 한 것 vs 남은 것

2026 프로덕션 RAG 체크리스트의 핵심 항목 대비 우리 상태:

| 프로덕션 요구(업계) | 우리 상태 |
|---|---|
| 과권한 서비스계정 방지(사용자별 ACL) | ✅ 검색 시점 ACL pre-filter |
| 인용 정확도 95%+ / faithfulness 0.85+ | ✅ 인용 검증 + 게이트(임계는 실측 필요) |
| 부재 질의 환각 방지 | ✅ abstention 단독 거부권 |
| 체크섬 게이트 증분 재인덱싱 | ✅ content_hash 증분 인덱싱 |
| 감사 로그(불변·추적) | ✅ trace·egress 감사(영속화 필요) |
| 평가 프레임워크(회귀 게이트) | ✅ 자체 하니스 + CI(골드셋 필요) |
| 시크릿 관리(KMS/Vault) | ⬜ 미구현 |
| GPU 오토스케일(큐 깊이 기반) | ⬜ 미구현 |
| 실제 모델·디코더 연결 | ⬜ 미구현(어댑터만) |

요지: **설계·로직 층은 2026 모범 사례를 상당 부분 선제 반영**했다. 남은 건 대부분
"실제 외부 시스템 연결"과 "운영 인프라"다 — 코드 정확성이 아니라 환경·데이터 문제.

## 1. 최우선 — 실제 어댑터 연결 (서비스의 전제)

### 1.1 HWP 디코더 — 리스크가 줄었다 (조사 성과)

§A 최대 리스크였던 HWP 파싱에 **실제 사용 가능한 라이브러리가 등장**했다(2026):

- **`hwp-hwpx-parser`** (PyPI, 2026.01) — 우리 `HwpDecoder` 인터페이스와 거의 일치.
  `text`, `tables`, `file_type`, `is_encrypted`, `is_valid`, `get_tables_as_markdown/csv`
  제공. HWP5·HWPX 모두 지원. **우리 어댑터에 바로 끼울 후보.**
- **`kordoc`** (GitHub, 2026.05) — 우리가 설계한 것을 이미 구현: 한국 공문서
  key-value 테이블 자동 감지, 한글 어절 끊김 복원, 머리글/바닥글 감지, 중첩 테이블 마커.
  HWP3/5·HWPX·HWPML·PDF·Office → Markdown. MCP 서버 통합.
- **한컴 공식** tech.hancom.com — HWP/HWPX 파싱 공식 가이드(포맷 구조 + Python).

**작업:** 위 라이브러리를 `StructuringParser`의 디코더 어댑터로 래핑 →
RawBlock 변환 → 기존 6개 파서 테스트가 실제 디코더로도 초록인지 확인.
복원율은 여전히 PoC로 실측해야 하지만, **"라이브러리가 없다"는 리스크는 해소**됐다.

### 1.2 PoC 실행 (Go/No-Go — 여전히 최우선)

- 실제 행정문서 표본 수십~수백 건 확보(층화: 포맷·연도·표 복잡도별)
- 1.1 디코더로 파싱 → `eval/poc.py`에 넣어 표 복원율 측정
- 최악 층·abstention 거부권 포함 GO/CONDITIONAL/NO_GO 판정
- **이 판정 전까지 실전 정확도는 미지수** — 서비스 약속을 하면 안 됨

### 1.3 임베딩·리랭커 모델 (self-host GPU)

- 한국어 임베딩 후보 벤치마크(KURE, BGE-M3 등) — 본 코퍼스 recall@k로 선정
- TEI/vLLM류 추론 서버에 모델 적재 → `embedding-server`/`reranker-server` 컨테이너
- `embedder.py`의 `EmbeddingModel`·`Morph` 어댑터를 실제 서버 HTTP 호출로 구현
- 형태소: Kiwi 통합(sparse 토큰화)

### 1.4 외부 LLM 연결

- 프로바이더 선정(한국어 성능·약관·리전·비용)
- `llm_client.py`의 `Transport` 어댑터를 실제 HTTP(httpx) 호출로 구현
- API 키는 **시크릿 매니저에서 주입**(2절) — 코드·이미지에 절대 넣지 않음

## 2. 운영 인프라 — 서비스로 뜨기 위한 것

2026 K8s LLM 서빙 모범 사례 기반.

### 2.1 시크릿 관리 (보안 필수)
- **base64 K8s Secret만으로는 부족.** External Secrets Operator로 AWS Secrets
  Manager/HashiCorp Vault/KMS에서 동기화.
- 대상: LLM API 키, JWT 시크릿, DB 자격증명, 오브젝트 스토리지 키.
- 현재 `settings.py`가 환경변수로 외부화돼 있어 주입 지점은 준비됨.

### 2.2 GPU 오토스케일
- HPA의 CPU 트리거는 LLM에 부적합 — **KEDA로 큐 깊이·KV캐시 사용률 기반 스케일**.
- GPU 노드 풀 + Cluster Autoscaler/Karpenter(노드 프로비저닝).
- MIG로 GPU 분할(임베딩·리랭커가 풀 GPU 불필요 시 비용 절감).
- 추론 서버 OOM 방지: `/dev/shm` 공유 메모리 볼륨, 필요시 양자화(AWQ).

### 2.3 네트워크 격리
- NetworkPolicy로 추론 파드 접근 제한 — API 게이트웨이·모니터링만 도달 가능.
- 외부 LLM egress는 명시적 허용 목록(데이터 유출 경계, v4).

### 2.4 헬스 프로브
- startup/readiness/liveness 분리. 현재 `/health`는 얕은 체크 →
  `/health/ready`에 의존성(Qdrant·DB·추론서버) 깊은 체크 추가(K8s readiness용).

### 2.5 무중단 배포
- 모델 업데이트가 in-flight 요청을 끊지 않도록 graceful shutdown.
- 카나리 롤아웃(새 모델·프롬프트를 일부 트래픽에 먼저).

## 3. 데이터·평가 — 정확도를 약속하려면

### 3.1 골드셋 구축 (측정의 토대 — 사람 투자 필요)
- 유형별 층화(body/table/multi_clause/absent/conflicting), 유형별 최소 표본
- 질의 3출처(실사용 로그·LLM 생성·적대적 수작업)
- 라벨러 간 일치도(IAA) 측정, 모호 질의 제외
- **이건 코드가 아니라 도메인 담당자의 라벨링 시간**

### 3.2 평가 프레임워크 선택 (조사: 셋 중 하나만)
2026 표준 셋 — 팀 성격에 맞는 하나만 쓴다(여러 개는 유지보수 부담):
- **Promptfoo** — YAML·CLI 중심, 프롬프트/검색설정을 코드로 다루는 팀
- **DeepEval** — pytest 호환, 파이썬 중심 팀(우리 자체 하니스와 결 유사)
- **RAGAS** — RAG 전용(faithfulness·context recall이 1급 시민)
- 우리 `eval/harness.py`와 병행하거나, 위 중 하나로 표준화 검토

### 3.3 SLO 실측·동결
- 모든 임계(현재 플레이스홀더)를 부하·평가로 실측 후 사업 합의로 동결
- faithfulness 0.85+가 규제 환경 베이스라인(조사)
- 부하 테스트: `eval/perf.py` 방법론(워밍업 제외·p95/p99·단계분해)로
  benchmark_serving류 도구와 함께 실측

## 4. 거버넌스·컴플라이언스 (행정문서 특성상 중요)

조사: 2026 RAG는 NIST AI RMF·EU AI Act 고위험 통제에 매핑되는 추세.

- **감사 로그 영속화** — 현재 trace는 인메모리. 모든 질의·검색결과·egress를
  불변 저장소에 영속화(규제 대응·사후 추적). 현재 `tracing.py`가 구조는 갖춤.
- **데이터 주권** — 행정문서 리전 국내 제약, 외부 LLM 약관·리전 법무 검토.
- **접근 감사** — 누가 무엇을 언제 조회했는가(ACL 결정 로그).
- **보존·삭제 정책** — 문서·로그 보존 기간, 파기 절차.

## 5. 권장 순서 (의존성 기반)

```
[1] HWP 디코더 어댑터 연결(1.1) ─┐
                                  ├─→ [2] PoC 실행·판정(1.2) ★Go/No-Go
[실제 행정문서 표본 확보] ────────┘
        ↓ (GO일 때만)
[3] 임베딩·리랭커·LLM 어댑터 연결(1.3, 1.4) + 시크릿 관리(2.1)
        ↓
[4] 골드셋 구축(3.1) → 평가·SLO 실측·동결(3.2, 3.3)
        ↓
[5] K8s 운영 인프라(2.2~2.5) + 감사 영속화(4)
        ↓
[6] 부하 테스트 → 카나리 → 프로덕션
```

핵심: **[2] PoC가 관문이다.** GO 전까지 3~6은 투자 대비 위험이 크다.
디코더 라이브러리가 생겨 [1]의 리스크는 줄었지만, 실제 표본 복원율은 여전히 미지수.

## 6. 한 줄 요약

코드(로직·계약·방어)는 2026 프로덕션 모범 사례를 상당 부분 선제 반영했다.
남은 것은 대부분 **외부 연결(디코더·모델·LLM)·운영 인프라(시크릿·GPU 스케일·감사)·
실제 데이터(표본·골드셋)**이며, 코드 정확성 문제가 아니다.

그리고 변하지 않는 결론 — **실제 HWP 표본으로 PoC를 통과하기 전까지,
이 시스템이 실전 정확도를 낼지는 GO도 NO-GO도 아니다.** 다행히 2026년 들어
쓸 만한 HWP 파싱 라이브러리가 등장해, 그 검증을 시작할 도구는 준비됐다.
