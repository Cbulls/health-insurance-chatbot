# 한글 행정문서 RAG 챗봇 — 설계 보고서 (v4)

**범위:** 요구사항 분석부터 코드 개발 직전(국면 A~D)까지
**대상 독자:** 시니어 엔지니어 / 기술 리더
**전제 제약:** HWP 포함 한글 행정·사무 문서, **클라우드(온프레미스 아님)**, **외부 API 허용** — 다룰 문서에 외부 API(학습 비사용 약관)로 보내면 안 되는 민감 등급은 없음(전부 외부 API 처리 가능)

---

## 변경 이력

**v1 → v2:** 폐쇄망(온프레미스) + 외부 API 금지 전제가 해제됐다. 영향 범위를 가두고 제약과 무관했던 설계는 보존하는 것이 변경 관리의 핵심 원칙이다.

- **폐기되지 않은 것 (64개 코드 계약 전부 유효):** 국면 A(HWP 파싱 상한), 국면 B 전체(스키마·경계·ACL·동시성), 국면 C(평가 하니스), VectorDB(Qdrant). HWP는 클라우드에서도 안 열리고, 데이터 계약·빈 창 레이스·평가 편향은 인프라 위치와 무관하다. 이것이 "막을 수 있는 건 코드로 증명한다"가 인프라 변경에 강건했다는 증거다.
- **변경된 것:** 국면 D 용량·토폴로지(§6), NFR-5 보안 — 경계 기반에서 데이터 등급 기반으로(§3).
- **새로 생긴 것:** 외부 유출 검증 경계(생성 직전), 보안등급 기반 생성 경로 라우팅, 네트워크 경계 관심사.

**v2 → v3:** "모델 선택"을 별도 기능 섹션에서 제외했다. 어떤 모델을 쓸지(KURE vs BGE-M3, API vs self-host)를 비교·선정하는 것은 본 설계 보고서의 범위가 아니라 구현 시점의 벤치마크 결정으로 미룬다. 단, 거기 섞여 있던 보안등급 기반 생성 경로 라우팅은 보안 설계의 핵심이므로 §3(보안)으로 흡수해 유지했다.

**v3 → v4:** 데이터 분류 결과, **다룰 문서에 외부 API로 보내면 안 되는 민감 등급이 없음**이 확인됐다. 이로써 설계가 단일 외부 API 경로로 단순화된다.

- **제거되는 것:** self-host LLM(GPU 용량·비용 부담 소멸), 보안등급 기반 생성 경로 라우팅(가를 경로가 하나뿐), 외부 유출 검증 경계(막을 유출 시나리오가 없음 — 존재하지 않는 위협에 대한 방어는 짓지 않는다). 국면 D의 self-host LLM GPU 용량 산정도 불필요.
- **그래도 남는 보안:** 사용자 단위 ACL(B-4)은 데이터 등급과 무관한 별개 문제 — "이 문서가 외부로 나가도 되는가"가 아니라 "이 사용자가 볼 권한이 있는가" — 이므로 그대로 유효. 외부 API 약관(학습 비사용·보존기간·리전) 검토는 여전히 §7의 경계.
- **이번에도 폐기되지 않은 것:** 국면 A·B·C·VectorDB의 64개 코드 계약. 인프라·모델 전략이 또 바뀌어도 데이터 계약과 평가 편향은 불변.

v4에서 바뀐 섹션은 §3(보안)·§6(국면 D)과 본 변경 이력이다.

## 0. 요약 (Executive Summary)

이 프로젝트는 한글 행정문서를 파싱·청킹해 VectorDB에 적재하고, LLM으로 질의응답하는 RAG 챗봇이다. 본 보고서는 **코드 첫 줄을 작성하기 전까지** 수행한 설계를 다룬다.

핵심 판단은 하나다. **이 프로젝트의 난이도는 LLM이 아니라 HWP 파싱과 한국어 검색에 몰려 있고, 성패는 평가셋을 얼마나 일찍 만드느냐에서 갈린다.** 인프라 제약이 풀려도 이 판단은 변하지 않는다 — HWP와 한국어 검색은 클라우드·API로 해결되는 문제가 아니기 때문이다.

작업은 네 국면으로 진행했다. (A) HWP 파싱 상한을 실측해 프로젝트 가능 여부 자체를 판정하는 PoC 설계, (B) 데이터·컴포넌트·동시성·접근통제를 코드 계약으로 동결, (C) 평가·관측 인프라를 제품 코드보다 먼저 구축, (D) 용량 산정과 서빙 토폴로지 확정. 그 사이 VectorDB를 Qdrant로 확정했다.

관통하는 설계 철학은 두 문장이다. **막을 수 있는 것은 코드로 증명한다. 막을 수 없는 것은 막힌다고 포장하지 않고 경계 밖으로 명시한다.**

## 1. 요구사항 분석 [v1과 동일]

### 2.1 기능 요구사항(FR)의 무게중심

일반적인 RAG 튜토리얼(평문 파싱 → 고정 청킹 → dense 검색)이 한글 행정문서에서 실패하는 지점을 FR의 중심에 놓았다.

- **FR-0 처리 경로 분기:** 인덱싱 여부는 포맷이 아니라 **양**이 결정한다. 컨텍스트에 들어가는 소량은 직주입, 초과 시 RAG.
- **FR-2 파싱(포맷별 라우팅):** "단일 포맷 통일"은 안티패턴. 통일한다면 PDF가 아니라 구조화 텍스트로. 파싱 출력은 평문이 아니라 **구조 보존 IR**.
- **FR-3 청킹:** 토큰 고정 분할이 아니라 **구조 기반 청킹** 우선. 토큰 분할은 단서 조항을 찢는다.
- **FR-5/6 검색:** 단일 dense 금지. **하이브리드(BM25+dense)+RRF** 기본값. 한국어 BM25는 형태소 분석기 전제.
- **FR-8 생성:** "문서에만 근거, 없으면 모른다"를 한국어로 명시. 인용 강제 + 인용 ID 코드 검증.

### 2.2 비기능 요구사항(NFR)의 무게중심

RAG에 고유한 세 무게중심: ① retrieval 시점 접근통제, ② 품질을 회귀 테스트로 박는 평가 게이트, ③ 재인덱싱을 전제로 한 설계.

### 2.3 환각의 두 실패 모드 분리

**Retrieval 실패형**(틀린 청크)은 검색 품질·임계값으로, **Context Unfaithfulness형**(컨텍스트 무시)은 인용 검증·생성 후 검증으로 대응. temperature 0은 환각 대책이 아니다.

## 2. Self-Critique — 설계의 약점 직시 [v1과 동일]

1. **HWP 파싱이 단일 실패점(SPOF):** 파싱 상한 측정을 요구사항으로 박지 않은 것이 가장 큰 잘못. → 국면 A.
2. **청킹이 파싱에 곱셈으로 종속:** 폴백 발동률이 시스템 건강 지표여야 함.
3. **평가 골드셋의 자기 회피:** LLM 생성 질의는 시스템이 잘 푸는 질의로 편향. → 국면 C.
4. **retrieval 시점 ACL의 취약성:** 두 인덱스 필터 불일치, 추론 누수. → B-4.
5. **upsert 레이스:** 전량 삭제·재적재 사이의 빈 창. → B-3.
6. **가장 깊은 약점은 기술 밖:** 사용자는 "판단"을 기대하나 RAG는 "검색·요약"만. 제품 범위로 닫아야 함.

## 3. 보안(NFR-5) 재설계 — 단일 외부 API 경로 + 사용자 ACL [v4 단순화]

폐쇄망 해제 + 민감 문서 부재로, 보안 설계가 두 축으로 정리된다. 데이터 등급 기반 라우팅은 더 이상 필요 없지만(가를 경로가 하나뿐), 사용자 접근통제는 그대로 남는다.

### 3.1 데이터 등급 기반 라우팅은 불필요 [v4 제거]

v2/v3는 민감 문서를 self-host로, 공개·내부 문서를 외부 API로 가르는 라우팅을 설계했다. 데이터 분류 결과 외부로 보내면 안 되는 민감 등급이 없음이 확인되어, **모든 문서가 단일 외부 API 경로로 처리된다.** 가를 경로가 하나뿐이므로 라우팅도, 생성 직전 외부 유출 검증 경계도 짓지 않는다. 존재하지 않는 위협에 대한 방어를 짓는 것은 부채다.

### 3.2 사용자 ACL은 그대로 유효 (등급과 무관한 별개 문제)

"문서가 외부로 나가도 되는가"(데이터 등급, 이제 전부 OK)와 "이 **사용자**가 이 문서를 볼 권한이 있는가"(접근통제)는 다른 문제다. 부서 A 직원이 부서 B 전용 문서를 검색으로 끌어오면 안 되는 것은 외부 API를 쓰든 말든 똑같이 막아야 한다. 따라서 **B-4의 retrieval 시점 ACL(누수 A·B 방어)은 외부 API 단일 경로에서도 그대로 유효**하다. 사용자 권한 필터는 Qdrant 검색 시점에 적용되고, 멀티턴 권한 재평가도 유지된다.

### 3.3 남는 위험 (정직하게)

외부 API는 제공자의 데이터 정책에 의존한다 — 학습 비사용 보장, 데이터 보존 기간, 리전(데이터 주권). "전부 외부 API 가능"이라는 판단의 근거가 이 약관 검토라면, 그 검토가 문서로 남아야 한다. 행정문서는 데이터 주권이 걸릴 수 있어 리전 제약(국외 리전 금지 등)이 필요할 수 있다. 코드가 아니라 계약·규정 검토 영역이며 법무·보안팀 합의가 전제다.

## 4. 국면 A — HWP 파싱 상한 PoC (Go/No-Go 판정) [v1과 동일]

목적은 **"이 프로젝트가 가능한가"에 숫자로 Go/No-Go를 내리는 것**. 이 게이트를 통과 못 하면 국면 B로 진행하지 않는다.

측정 계획의 네 원칙: ① 임계를 측정 전에 동결(굿하트 방어), ② 무작위가 아닌 층화 표집(최악 케이스 강제 포함), ③ 파싱 품질 × end-to-end 정확도를 곱셈으로 연결, ④ abstention 단독 거부권. 판정은 GO/CONDITIONAL/NO-GO 3구간.

## 5. 국면 B — 설계 동결 (코드 계약) [v1과 동일]

핵심 원칙: **불변식은 주석이 아니라 실행되는 코드다. 위반 객체는 생성 자체가 불가능해야 한다.** 직렬화 형식은 Pydantic(스키마·검증·불변식·JSON Schema 자동생성을 한 객체에).

### 5.1 B-1 두 스키마

IR 스키마: 표를 문자열 평탄화하지 않고 **셀 좌표 구조로 보존**(마크다운은 파생물). 메타데이터 스키마: 모든 필드를 "빼면 죽는 기능"으로 역산, 두 fail-closed 불변식(`acl_tags` null 불허, `source_block_ids` 1개 이상). 계보 체인(IR block_id → source_block_ids → 인용)이 신뢰성의 척추. **v4 주: `security_level` 필드는 단일 API 경로에서 생성 라우팅 용도로는 쓰이지 않으나, 향후 등급 정책이 생길 때를 위해 스키마에는 유지한다(제거 비용 > 보존 비용).** 테스트 15개 통과.

### 5.2 B-2 컴포넌트 경계

다섯 경계가 방어선. 경계1(PoC 임계 런타임 게이트), 경계2(M3 교차검증), 경계3(stale 벡터·반쪽 하이브리드 거부), 경계4(권한 필수 인자), 경계5(위조 인용 차단·abstention). **v4 주: v2/v3에서 추가하려던 외부 유출 검증 경계는 민감 문서 부재로 불필요해져 짓지 않는다.** 테스트 17개 통과.

### 5.3 B-4 ACL 일관성

누수 A(단일 필터 명세로 드리프트 차단), 누수 B(pre-filter 1차선), 누수 C(추론 누수는 원천 불가 — 제품 정책 영역). 멀티턴 권한은 매 턴 재평가. 테스트 7개 통과.

### 5.4 B-3 동시성

**버전 태깅 + 원자적 포인터 전환**으로 빈 창 제거. 트랜잭션(엔진 의존)이 아니라 애플리케이션 레벨 장치를 택해 엔진 능력에 베팅하지 않음. 스레드 스트레스: 순진한 방식 47% 빈 창 → 우리 해법 0.

## 6. 국면 D — 용량 산정과 서빙 토폴로지 [v4 단순화]

### 6.1 LLM 용량 산정이 사라진다

폐쇄망의 1번 제약 "고정 GPU"가 풀리고, 민감 문서 부재로 self-host LLM까지 사라지면서, **LLM 용량 산정이 통째로 단순해진다.**

- **LLM(외부 API 단일 경로):** GPU 용량 산정이 **사라진다.** 동시성은 제공자 rate limit, 비용은 토큰 단가로 결정된다. `capacity.py`의 LLM 부분에서 `MeasuredLLM`(GPU 처리량·KV캐시)이 `rate_limit_rpm`·`cost_per_1k_tokens`로 대체된다. **자기참조 버그로 고생한 GPU 동시성 계산이 LLM에 대해선 완전히 불필요해진다.** 대신 새 관심사: API 지연 변동성(p99가 외부 요인에 흔들림), rate limit 백오프·큐잉, 토큰 비용 폭주 방어(쿼터·상한).
- **임베딩·리랭커:** self-host 한국어 특화를 기본으로 두면 GPU 산정이 남지만, 클라우드라 **오토스케일** 가능 → 폐쇄망의 "고정 대수"가 "기준 용량 + 버스트"로. 인덱싱 배치 대량 임베딩을 스케일아웃으로 흡수, NFR-2 읽기/쓰기 격리가 쉬워진다. (임베딩까지 외부 API로 가면 이 GPU 산정도 사라지나, 전체 코퍼스 외부 전송·한국어 우위·재임베딩 락인 때문에 self-host 기본이 무난 — 구현 시 결정.)

**비용 모델:** LLM은 변동비(API 토큰), 임베딩·리랭커는 고정비(self-host GPU) 또는 변동비(API). 질의당 비용 관측은 LLM 토큰 사용량이 지배 변수.

### 6.2 서빙 토폴로지

폐쇄망 네 영역 중 셋이 단순해진다.

- **사라지는 것 — 에어갭 모델 반입:** 단방향 매체·서명 재검증 절차 불필요. 클라우드 정상 배포 파이프라인으로 대체. 단, **임베딩·리랭커 교체 시 평가셋 재통과 게이트는 유지**(품질 절차라 인프라와 무관).
- **단순해지는 것 — 추론 계층:** v2/v3의 "혼합 추론 계층(API + self-host LLM)"이 **외부 LLM API 커넥터 단일 경로**로 줄어든다. self-host LLM GPU가 사라진다. 임베딩·리랭커만 self-host GPU(오토스케일)로 남는다. 보안등급 라우팅 분기도 사라진다.
- **그대로인 것:** 게이트웨이 단독(**사용자 ACL** 해결, B-4), 읽기/쓰기 분리(NFR-2), Qdrant(self-host or Cloud, 검증 기능 동일).
- **네트워크 경계:** LLM 트래픽이 외부 API로 나간다. 유출 지점이 단일하므로 ① API 키 관리, ② 나가는 트래픽 로깅·감사(어떤 질의·컨텍스트가 외부로 갔는지), ③ 약관 준수(§3.3 리전·보존)가 여기 집중. 민감 문서 차단 검증은 불필요(민감 문서 없음).

## 8. 의도적으로 닫지 않은 것 (정직한 경계)

- **추론 누수(누수 C):** 청크 단위 ACL로 원천 불가. 제품 정책 영역.
- **기대-역량 격차:** 사용자는 법령 "판단"을 기대하나 RAG는 "검색·요약"만. 제품 범위·면책으로.
- **[v2] 외부 API 데이터 정책 의존:** 학습 비사용·보존기간·데이터 주권은 제공자 계약·법무 검토 영역. 코드로 못 박음.
- **[v2] 데이터 주권:** 국외 리전 API 사용 가부는 규정 검토 필요.
- **LLM-as-judge 채점자 신뢰도:** 별도 작업.
- **골드셋의 양과 대표성:** 도메인 담당자 라벨링 투자에 의존.
- **실측 용량 숫자:** 계산기는 측정 프로토콜이지 결과가 아님. 실환경 부하 테스트로 보정.
- **[v3] 모델 선정:** 어떤 임베딩·LLM·리랭커를 쓸지(KURE vs BGE-M3, 어떤 API LLM 등)의 비교·선정은 본 설계 범위 밖. 구현 시점에 본인 코퍼스 벤치마크로 결정.
- **[v4] 향후 민감 등급 도입:** 지금은 민감 문서가 없어 단일 API 경로지만, 추후 민감 등급이 생기면 `security_level` 기반 라우팅과 외부 유출 검증 경계를 다시 도입해야 한다. 스키마에 `security_level`을 남겨둔 이유.
- **HA/이중화 물리 배치:** 하드웨어/클라우드 리전 확정 후 결정.

## 9. 결론 및 다음 단계

인프라 제약(폐쇄망, 외부 API 금지)이 풀리고 민감 문서 부재가 확인되면서, 설계가 **단일 외부 API 경로로 단순해졌다.** 그럼에도 **국면 A·B·C·VectorDB의 64개 코드 계약은 한 줄도 폐기되지 않았다.** 인프라·모델 전략이 세 번 바뀌는 동안(v1→v4) 데이터 계약·동시성·평가 편향은 불변이었다 — "막을 수 있는 건 코드로 증명한다"는 방법론이 변경에 강건했다는 누적 증거다.

v4의 단순화: ① 보안 — 데이터 등급 라우팅·외부 유출 검증 경계 제거(가를 경로·막을 위협이 없음), **사용자 ACL은 유지**(등급과 무관한 별개 문제), ② 용량 — self-host LLM 제거로 LLM GPU 산정이 통째로 rate limit·비용 모델로 대체, ③ 토폴로지 — 혼합 추론 계층이 단일 API 커넥터로 축소(임베딩·리랭커만 self-host GPU 잔존). 제거의 정당성은 "존재하지 않는 위협에 대한 방어는 짓지 않는다"이다.

설계 과정에서 두 번, 검증 테스트가 설계자 자신의 버그를 잡았다(B-1 order_index 전순서, 국면 D 동시성 자기참조). "계약을 코드로 박는다"가 실제로 작동한다는 증거다.

**가장 중요한 운영 권고는 모든 판본에서 변하지 않는다 — 국면 A의 HWP 파싱 PoC를 가장 먼저 수행하라.** 인프라가 클라우드로 바뀌고 외부 API가 열리고 추론 계층이 단순해져도 HWP는 여전히 안 열린다. 이 Go/No-Go 없이 본 개발에 착수하는 것이, 네 판본의 보고서가 일관되게 경계한 단 하나의 실수다.
