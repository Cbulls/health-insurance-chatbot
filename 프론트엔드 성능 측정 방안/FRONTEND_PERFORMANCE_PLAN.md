# harag — 프론트엔드 성능 측정 설계 (Frontend Performance Plan)

이 문서는 harag 챗봇 **프론트엔드**의 성능을 측정하는 방법을 설계한다.

> **전제:** 현재 프로젝트는 백엔드(API)만 있고 프론트엔드는 아직 없다.
> 따라서 이 문서는 "기존 코드 측정법"이 아니라 **프론트엔드를 만들 때 적용할 측정 기준·설계**다.
> 백엔드는 `StreamEvent`(token/abstain/citations/done)로 스트리밍을 내보내므로,
> 프론트는 이 이벤트 스트림을 소비해 렌더링한다 — 측정도 이 구조를 전제로 한다.

핵심 인식 — **RAG 챗봇의 체감 성능은 페이지 로딩이 아니라 "응답이 흐르는 경험"이 좌우한다.**
일반 웹 지표(LCP 등)만으로는 부족하고, **스트리밍 고유 지표**가 중심이어야 한다.

---

## 1. 측정의 두 축

프론트엔드 성능은 성격이 다른 두 축으로 나뉜다. 섞으면 안 된다.

| 축 | 무엇 | 대표 질문 |
|---|---|---|
| **A. 로딩 성능** | 앱이 뜨고 상호작용 가능해지기까지 | "챗봇 화면이 얼마나 빨리 뜨나?" |
| **B. 런타임/스트리밍 성능** | 질문 후 응답이 흐르는 경험 | "답이 얼마나 빨리·매끄럽게 나오나?" |

챗봇은 **한 번 로딩 후 오래 머무는 앱**이라, B축(런타임)이 A축(로딩)보다 체감에 더 중요하다.

---

## 2. A축 — 로딩 성능 (Core Web Vitals)

표준 웹 지표. 측정 도구: Lighthouse, WebPageTest, 그리고 실사용자 측정(RUM)은 `web-vitals` 라이브러리.

| 지표 | 의미 | 권장 기준(good) |
|---|---|---|
| **LCP** (Largest Contentful Paint) | 주요 콘텐츠 표시 | ≤ 2.5s |
| **INP** (Interaction to Next Paint) | 상호작용 반응성(FID 후속) | ≤ 200ms |
| **CLS** (Cumulative Layout Shift) | 레이아웃 흔들림 | ≤ 0.1 |
| **FCP** (First Contentful Paint) | 첫 픽셀 | ≤ 1.8s |
| **TTI** (Time To Interactive) | 입력창이 실제로 쓸 수 있게 됨 | ≤ 3.8s |

**챗봇 특화 주의:**
- 입력창(질문 박스)이 **가장 먼저 인터랙티브**해야 한다 — TTI를 입력창 기준으로 본다.
- 대화 이력 로딩이 CLS를 유발하지 않게(메시지가 끼어들며 화면이 밀리면 안 됨).

---

## 3. B축 — 스트리밍/런타임 성능 (챗봇의 핵심)

여기가 RAG 챗봇 고유 영역이다. 백엔드 `StreamEvent` 소비 경험을 측정한다.

### 3.1 핵심 지표

| 지표 | 정의 | 왜 중요한가 |
|---|---|---|
| **TTFT** (Time To First Token) | 질문 전송 → 첫 token 이벤트 도착 | **체감 반응성의 1순위.** "멈춰있다"는 느낌을 가르는 선 |
| **토큰 스트리밍 레이트** | 초당 도착 토큰 수, 토큰 간 간격 분포 | 끊김(stall) 없이 매끄럽게 흐르는가 |
| **스트리밍 stall** | 토큰 간 간격이 임계(예 500ms) 초과한 횟수 | 끊겨 보이는 구간 — 체감 악화의 주범 |
| **E2E 완료 시간** | 질문 → done 이벤트 | 전체 응답 소요 |
| **abstention 응답 시간** | 질문 → abstain 이벤트 | "모른다"는 빨라야(LLM 미호출) — 별도 집계 |
| **인용 렌더 시간** | done 후 citations 표시까지 | 출처가 늦게 뜨면 신뢰 저하 |

### 3.2 TTFT의 분해 (프론트 vs 백엔드)
TTFT = **네트워크 왕복 + 백엔드 처리(재작성~생성 첫 토큰) + 프론트 렌더 시작**.
백엔드 TTFT(성능 측정 문서 §2.1)와 분리해서 봐야 한다:
- 프론트 TTFT - 백엔드 TTFT = **네트워크 + 클라이언트 오버헤드**
- 이 차이가 크면 프론트(연결 설정·파싱·렌더)의 문제, 작으면 백엔드/LLM 문제.

### 3.3 렌더링 성능 (스트리밍 중)
토큰이 흘러들어올 때 **매 토큰마다 리렌더하면 프레임 드랍**이 난다.
- **긴 응답 중 프레임률(FPS)** — 60fps 유지하는가, 긴 답변에서 떨어지는가
- **메인 스레드 블로킹** — 마크다운 파싱·구문 강조가 긴 작업(long task)을 만드는가
- **DOM 노드 증가** — 긴 대화에서 노드가 무한정 쌓여 느려지는가(가상 스크롤 필요성)

## 4. 측정 방법 — Lab vs Field

### 4.1 Lab (합성, 통제된 환경)
재현 가능한 회귀 감시용.
- **Lighthouse CI** — 빌드마다 A축 지표 측정, 임계 미달 시 알림
- **Playwright/Puppeteer 스크립트** — 질문 전송 → 첫 토큰 → 완료를 자동 측정(B축)
  - `performance.now()`로 전송 시각 기록 → 첫 `token` 이벤트 수신 시각 → TTFT
  - 토큰 이벤트 타임스탬프 배열 → 간격 분포·stall 카운트
- **네트워크 스로틀링** — Fast 3G/4G/저사양 CPU로 약한 환경 재현(행정망 단말 고려)

### 4.2 Field (실사용자 측정, RUM)
실제 사용자 환경의 분포. Lab은 한 환경, Field는 천차만별.
- `web-vitals` 라이브러리로 LCP/INP/CLS 수집 → 분석 백엔드 전송
- **커스텀 RUM 이벤트:** TTFT, 스트리밍 레이트, stall을 직접 계측해 전송
  ```js
  const t0 = performance.now();
  // 첫 token 이벤트 수신 시
  reportMetric('ttft', performance.now() - t0);
  ```
- **백분위로 본다** — 평균이 아니라 p75/p95(백엔드 측정과 같은 원칙). 느린 사용자가 진짜 문제.

---

## 5. 측정 항목 — 기능/비기능 대응

### 5.1 기능적 정확성 (성능과 별개로 측정)
- 스트리밍 중 **토큰 유실·순서 뒤바뀜 없는가** (SSE/WebSocket 신뢰성)
- abstain 이벤트 시 **"모른다" UI가 정확히 표시**되는가
- **인용(citations)이 실제 출처와 일치**하게 렌더되는가
- 멀티턴에서 **대화 맥락이 UI에 정확히 유지**되는가
- 네트워크 끊김 시 **에러 상태·재시도 UI**가 동작하는가

### 5.2 비기능(성능) SLO 예시
측정 전 동결(굿하트 방어). 예시값이며 실측 후 확정:

| 지표 | 목표(p75) | 한계(p95) |
|---|---|---|
| 앱 로딩 LCP | ≤ 2.5s | ≤ 4s |
| 입력창 TTI | ≤ 2s | ≤ 3.5s |
| 질의 TTFT | ≤ 1.5s | ≤ 3s |
| 스트리밍 stall(>500ms 간격) | 0회 | ≤ 2회 |
| 스트리밍 중 프레임률 | ≥ 50fps | ≥ 30fps |
| INP(상호작용 반응) | ≤ 200ms | ≤ 500ms |

## 6. 챗봇 특화 측정 시나리오 (체크리스트)

- [ ] **콜드 로드** — 캐시 없이 첫 방문, 입력창 인터랙티브까지
- [ ] **TTFT** — 질문 전송부터 첫 토큰까지(여러 질의 유형으로)
- [ ] **긴 응답 스트리밍** — 긴 답변에서 stall·프레임 드랍 없는가
- [ ] **abstention 경로** — "모른다"가 빠르게 표시되는가
- [ ] **인용 렌더** — 출처가 적시에 정확히
- [ ] **멀티턴 누적** — 대화가 길어질 때 DOM 증가·스크롤 성능
- [ ] **약한 환경** — 저사양 CPU·느린 네트워크(행정망 단말)에서 사용 가능한가
- [ ] **동시 입력** — 응답 스트리밍 중 사용자가 새 입력/취소할 때 반응성
- [ ] **네트워크 단절** — 스트리밍 중 끊김 시 에러·재연결 UI
- [ ] **모바일/태블릿** — 반응형 레이아웃의 CLS·터치 반응성

---

## 7. 측정 인프라 설계

- **CI 연계:** Lighthouse CI를 PR 게이트(A축 회귀 차단), Playwright 스트리밍 측정은 야간.
- **RUM 파이프라인:** `web-vitals` + 커스텀 지표 → 수집 엔드포인트 → 대시보드(Grafana 등).
- **백엔드 trace 연계:** 프론트가 보낸 trace_id를 백엔드 `QueryTrace`와 연결 →
  **TTFT를 프론트·네트워크·백엔드로 완전 분해**(어느 구간이 느린지 한 번에 진단).
- **예산(Performance Budget):** 번들 크기·이미지·폰트에 상한을 두고 빌드 시 검사.

---

## 8. 한계 — 측정으로도 못 잡는 것 (정직하게)

- **실기기 다양성** — Lab은 대표 기기 몇 종. 실제 단말 파편화는 Field(RUM)로만 보임.
- **인지된 성능 ≠ 측정된 성능** — 스트리밍은 "느려도 흐르면" 빠르게 느껴진다. 정량 지표와 체감의 괴리는 사용성 테스트로 보완.
- **외부 LLM 변동성** — TTFT 꼬리는 외부 LLM 지연에 좌우됨(프론트 통제 밖). 분해 측정으로 책임 구간만 분리.
- **행정망 환경** — 실제 배포처의 네트워크·단말·브라우저(구형 포함) 제약은 현장 측정 필요.

---

## 요약

RAG 챗봇 프론트 성능은 세 질문에 답해야 한다:
1. **빨리 뜨는가?** — A축: LCP/INP/CLS/TTI (입력창 기준)
2. **빨리·매끄럽게 응답하는가?** — B축: **TTFT + 스트리밍 stall + 프레임률** (챗봇의 핵심)
3. **정확히 동작하는가?** — 토큰 무결성·인용 일치·에러 UI

핵심은 **TTFT와 스트리밍 매끄러움**이다. 백엔드 trace_id와 연계하면 지연을
프론트·네트워크·백엔드로 분해해 책임 구간을 정확히 짚을 수 있다.
프론트엔드 구현 시 이 지표들을 처음부터 계측에 넣는 것(observability by design)을 권한다.
