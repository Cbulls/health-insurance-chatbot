# API 게이트웨이 설계서

**컴포넌트:** `harag.api` — 시스템의 신뢰 경계이자 단일 진입점
**대상 독자:** API 소비자(프런트엔드·통합) + 게이트웨이 유지보수자
**상태:** 골격 구현 + 신뢰 경계 검증 완료(계약 테스트 6/6). 비즈니스 파이프라인은 주입 대기.

---

## 1. 게이트웨이의 위상 — 왜 단순한 라우터가 아닌가

이 컴포넌트는 "FastAPI 라우트 모음"이 아니라 **신뢰 경계(trust boundary)**다. 설계 보고서의 두 결정이 이를 강제한다:

- 보고서 §3: "ACL은 검색 전에 해결돼야 한다."
- 토폴로지: "게이트웨이가 단독으로 권한을 해결한다."

따라서 **게이트웨이를 통과한 뒤로 모든 하위 컴포넌트(검색·생성)는 "권한은 이미 해결됐다"를 전제**한다. 이 경계가 새면 그 아래 전부가 샌다. 게이트웨이 설계의 본질은 라우트가 아니라 세 책임의 계층 분리다:

1. **인증(Authentication):** 너는 누구냐 — 미들웨어/의존성에서 토큰 검증
2. **인가(Authorization/ACL):** 무엇을 볼 수 있냐 — `AuthContext`를 하위로 전파
3. **요청 검증(Validation):** 입력이 계약에 맞냐 — Pydantic

---

## 2. 설계 원칙 (모든 결정의 기준)

| # | 원칙 | 구현 |
|---|---|---|
| 1 | 신뢰 경계는 단일·명시적 | `require_auth` 의존성을 모든 보호 라우트에 강제. 누락이 구조적으로 불가 |
| 2 | 요청·응답은 Pydantic 계약 | 기존 스키마 재사용, 별도 검증 계층 없음 |
| 3 | 에러는 내부 구조를 누설 안 함 | 스택·내부 메시지는 로그에만, 응답엔 trace_id만 |
| 4 | 스트리밍이 1급 시민 | 답변 생성은 SSE(TTFT 단축) |
| 5 | 관측가능성 내장 | 모든 요청에 trace_id, 단계 전파 |

## 3. 인증·인가 — 신뢰 경계의 핵심

### 3.1 require_auth 의존성

`require_auth`는 FastAPI 의존성(`Depends`)으로 모든 보호 라우트에 주입된다. 이것이 신뢰 경계의 물리적 구현이다. B-2에서 `AuthContext`를 컴포넌트 필수 인자로 박은 것과 같은 정신 — **인증 없는 접근이 타입·구조 레벨에서 불가능**하다.

흐름: `Bearer 토큰 → IdP.verify() → AuthContext`. 반환된 `AuthContext`만이 하위(검색·생성)로 흐른다.

### 3.2 인증 백엔드는 추상화(미정 결정)

인증 방식(자체 JWT vs 기업 OIDC/SSO)은 코드 설계서 §9에서 미룬 결정이다. `IdentityProvider` Protocol 뒤에 두어, 방식이 정해지면 **게이트웨이 코드 변경 없이** 구현만 주입한다. 미설정 시 기본은 fail-closed(503) — 설정 안 된 IdP가 인증을 통과시키지 않는다.

### 3.3 fail-closed 재확인

IdP가 신원을 반환해도, **권한 태그(`acl_tags`)가 비어 있으면 403으로 차단**한다. B-4의 "acl_tags 비면 인덱싱 거부" 원칙을 게이트웨이에서도 강제 — 권한 범위 없는 신원은 아무것도 못 본다. (검증: 계약 테스트 "no scope -> 403".)

### 3.4 403 vs 404 — 누설 방지

권한 없는 리소스·엔드포인트는 **404로 통일**한다. 403("존재하지만 권한 없음")은 리소스의 존재 여부를 누설하기 때문이다. 행정 도메인에서 "그 문서는 있지만 당신은 못 본다"는 정보 자체가 민감하다. 예: 업로드 권한 없는 사용자의 `POST /v1/documents` → 404(이 엔드포인트의 존재를 알리지 않음).

## 4. 엔드포인트 계약

### 4.1 `POST /v1/query` — 질의

**요청**
```json
{ "query": "출장비 한도는?", "conversation_id": "optional-uuid" }
```
- `query`: 1~2000자(Pydantic 검증, 위반 시 422)
- 검색 깊이(top_k 등)는 **클라이언트가 못 정한다** — 서버 설정이 결정(남용·비용 방어)

**응답 200**
```json
{
  "answer": "국내출장 여비 한도는 1일 5만원입니다.",
  "citations": [
    {"citation_label": "여비규정 제12조", "source_document": "여비규정.hwp", "page": 12}
  ],
  "abstained": false,
  "abstain_reason": null,
  "trace_id": "..."
}
```

**abstention 응답**(답이 없거나 근거 부족 시)
```json
{ "answer": null, "abstained": true, "abstain_reason": "no_relevant_documents", "trace_id": "..." }
```
환각보다 abstention이 안전하다는 보고서 원칙의 API 표현. 클라이언트는 `abstained`로 분기한다.

**인증:** 필수(Bearer). 미인증 401, 권한범위 없음 403.

### 4.2 `POST /v1/query/stream` — 스트리밍 질의

SSE로 토큰을 흘린다(TTFT 단축, FR-8/NFR-1). 핵심 안전장치: **abstention은 스트리밍 시작 전에 결정**된다. 빈 컨텍스트면 토큰을 흘리기 전에 `abstain` 이벤트를 보내 — 지어낸 토큰이 사용자에게 노출되지 않는다.

이벤트 종류: `token`(부분 답), `citations`(출처), `abstain`(거부), `done`(종료).

### 4.3 `POST /v1/documents` — 문서 수집

**비동기**다(NFR-2 읽기/쓰기 격리). 업로드 → 즉시 `202 Accepted` → 실제 파싱·임베딩은 indexing-worker가 큐에서 처리. 라우트가 동기로 파싱하면 무거운 작업이 읽기 경로를 막는다.

- 멱등 키: 내용 SHA-256 해시 → 같은 문서 재업로드 시 중복 방지(FR-1)
- 크기 상한 100MB(남용 방지), 빈 파일 400
- 업로드 권한: 조회보다 강한 권한(`role:document-admin`) 요구, 없으면 404(누설 방지)

**응답 202**
```json
{ "document_id": "sha256-prefix", "status": "accepted", "trace_id": "..." }
```
`status`: `accepted`(신규) / `duplicate`(이미 있음). 실제 인덱싱 성공 여부는 별도 상태 조회로(파싱이 quarantine될 수 있음 — FR-1).

### 4.4 `GET /health`

인증 불필요. 얕은 헬스(프로세스 생존). 의존성(Qdrant·DB) 깊은 체크는 `/health/ready`로 분리 예정(K8s readiness probe용).

## 5. 에러 의미론

| 상태 | 의미 | 누설 방지 |
|---|---|---|
| 401 | 인증 필요/실패 | WWW-Authenticate 헤더만 |
| 403 | 인증됐으나 권한 범위 없음 | 리소스 존재는 404로 별도 처리 |
| 404 | 없음 **또는** 권한 없는 리소스 | 존재 여부 누설 차단 |
| 413 | 업로드 크기 초과 | — |
| 422 | 요청 스키마 위반 | Pydantic 자동(필드 단위) |
| 500 | 내부 오류 | **스택·내부 메시지는 로그에만**, 응답은 trace_id만 |
| 503 | 의존성 미설정/불가 | IdP 미설정 등 fail-closed |

**핵심(설계 원칙 3):** 500 응답은 `{"error": "internal error", "trace_id": "..."}`만 반환한다. 상세는 trace_id와 함께 로그에. 지원팀은 trace_id로 로그를 대조한다. 스택트레이스를 응답에 흘리는 것은 행정 시스템에서 보안 사고다.

## 6. 관측가능성 (NFR-6)

- **trace_id:** 요청마다 부여(또는 `X-Trace-Id` 헤더 승계), 응답 헤더·로그·하위 컴포넌트 trace에 전파. 환각 신고가 들어오면 trace_id로 검색→리랭킹→생성 전 단계를 재구성(보고서의 "단계별 단일 trace").
- **미들웨어 순서:** `SafeError`(최외곽) → `Trace`(안쪽) → 라우트. 이 순서라야 500 에러 응답에도 trace_id가 붙는다.
- 모든 응답에 trace_id 헤더 보장(검증: 계약 테스트 "trace header present").

---

## 7. 아키텍처 — 얇은 라우트

라우트는 **얇다**. 비즈니스 로직(검색·생성)은 컴포넌트(retriever, generator)에 있고, 라우트는 (1) 인증 강제 (2) 컴포넌트 연결 (3) 안전한 응답 변환만 한다.

- **의존성 주입(`deps.py`):** 라우트는 `QueryPipeline`·`IngestQueue` Protocol만 안다. 실제 구현은 앱 시작 시 주입 → 컴포넌트 교체·테스트 용이, 게이트웨이가 비즈니스 로직에 안 묶임.
- **응답 변환:** 내부 청크 메타(`embedding_model_id` 등)는 응답에서 떨군다. 사용자에겐 `citation_label`·`source_document`·`page`만(누설 방지).

이 분리의 이점: 게이트웨이(신뢰 경계)와 파이프라인(비즈니스 로직)을 독립적으로 테스트·배포·교체한다.

## 9. 의도적으로 미루는 것 (정직한 경계)

- **인증 방식 확정:** 자체 JWT vs 기업 OIDC/SSO — 조직 IdP 환경 확인 후. `IdentityProvider` 뒤에 끼움.
- **QueryPipeline·IngestQueue 구현:** 검색·생성·인덱싱 워커 조립 후 주입. 지금은 Protocol만.
- **Rate limiting:** API 게이트웨이 앞단(클라우드 LB/게이트웨이) 또는 미들웨어로. 외부 LLM rate limit(PERF-07)과 별개의 사용자 단위 제한.
- **세션/대화 상태 저장:** `conversation_id` 기반 멀티턴 상태를 어디 둘지(Redis 등)는 멀티턴 구현 시.
- **CORS·CSP 등 웹 보안 헤더:** 프런트엔드 도메인 확정 후.

---

## 10. 결론

게이트웨이는 **신뢰 경계**다. 핵심 설계 결정 셋: (1) `require_auth` 의존성으로 인증을 구조적으로 강제(누락 불가), (2) 권한범위 없는 신원을 fail-closed로 차단하고 누설을 404로 막음, (3) 라우트를 얇게 유지해 신뢰 경계와 비즈니스 로직을 분리. 모든 결정은 보고서의 "ACL은 검색 전에 해결" 원칙의 물리적 구현이며, 신뢰 경계 동작은 계약 테스트 6개로 고정돼 회귀를 막는다.
