# 마스터 설계서 — 한글 행정문서 RAG 챗봇 (harag)

**문서 성격:** 프로젝트 전체를 조망하는 마스터 설계서. 상세는 각 전문 문서로 위임(부록 A).
**독자:** 의사결정자(§0만), 신규 합류 엔지니어(전체), 컴포넌트 담당자(해당 섹션 + 링크).
**읽는 법:** §0~§4가 핵심(80%의 독자는 여기서 멈춰도 됨). §5 이하는 참조용.

---

## 0. 한 장 요약 (Executive Summary)

**문제.** 한글 행정문서(HWP 포함)를 기반으로 질문에 답하는 챗봇. 사용자가 "출장비 한도는?"이라 물으면 관련 규정을 찾아 출처와 함께 답한다(RAG — 검색 증강 생성).

**핵심 판단.** 이 프로젝트의 난이도는 LLM이 아니라 **HWP 파싱과 한국어 검색**에 있고, 성패는 **평가셋을 얼마나 일찍 만드느냐**에서 갈린다. 따라서 코드 전 단계의 목적을 "아키텍처 그리기"가 아니라 **"가장 치명적인 불확실성을 먼저 죽이는 리스크 소각"**으로 정의했다.

**기술 스택.** Python 3.12 + FastAPI + Qdrant(하이브리드 검색) + Docker/K8s + self-host GPU 임베딩 + 외부 API LLM.

**설계 철학.** 두 문장으로 관통한다 — **막을 수 있는 것은 코드로 증명한다. 막을 수 없는 것은 막힌다고 포장하지 않고 경계 밖으로 명시한다.**

**현재 상태.** 요구사항·계약·검증 완료(코드 계약 70개 + 동작 테스트 9개 통과). 검색기·생성기·게이트웨이 구현됨. **다음 작업: 국면 A HWP 파싱 PoC**(프로젝트 Go/No-Go 판정).

## 1. 배경과 목표

### 1.1 문제 정의

행정·사무 문서는 양이 많고 표·조항 구조가 복잡하며, 필요한 정보를 사람이 일일이 찾기 어렵다. 자연어 질의로 정확한 답과 출처를 받는 시스템이 목표다.

### 1.2 목표와 성공 기준

- 한글 문서(HWP/HWPX/PDF/DOCX) 기반 정확한 질의응답
- 답변에 검증 가능한 출처 제시
- 모르는 것은 지어내지 않고 "모른다"고 답함(abstention)
- 성공 기준: 사업이 동결한 정확도 SLO 충족(국면 A에서 역산)

### 1.3 비목표 (Non-Goals)

- **법령 해석·판단 대행:** 시스템은 "규정에 이렇게 쓰여 있다"까지. "그래서 당신 사안은 이렇게 처리된다"는 판단은 하지 않는다(제품 면책 영역).
- **문서 작성·편집:** 조회·질의응답만. 문서 생성은 범위 밖.
- **실시간 협업·워크플로:** 단일 질의응답에 집중.

### 1.4 제약

- 클라우드(온프레미스 아님), 외부 API 허용
- 다룰 문서에 외부 API로 보내면 안 되는 민감 등급 **없음**(데이터 분류 확인됨) → 단일 외부 API 경로
- 데이터 주권: 행정문서 → 리전 제약 가능(법무 검토 전제)

> 상세: 부록 A의 `design_report_v4.md` §1, 변경 이력(v1 폐쇄망 → v4 단일 API 경로)

## 2. 요구사항 요약

### 2.1 기능 요구사항 (핵심)

- **FR-0 경로 분기:** 인덱싱 여부는 포맷이 아니라 **양**이 결정(컨텍스트 초과 시만 RAG)
- **FR-2 파싱:** 포맷별 라우팅, 출력은 평문이 아니라 **구조 보존 IR**. 표는 셀 좌표로(평탄화 금지)
- **FR-3 청킹:** 토큰 고정이 아니라 **구조 기반**(조·항·호 우선). 단서 조항을 찢지 않음
- **FR-5/6 검색:** 단일 dense 금지. **하이브리드(BM25+dense)+RRF**. 한국어 BM25는 형태소 분석기 전제
- **FR-8 생성:** 문서에만 근거, 없으면 abstain. 인용 + **인용 ID 코드 검증**

### 2.2 비기능 요구사항 (무게중심)

RAG 고유의 세 무게중심: ① **retrieval 시점 접근통제**(보안), ② **품질을 회귀 테스트로 박는 평가 게이트**, ③ **재인덱싱 전제 설계**(유지보수성). 그 외 지연(p95/p99 SLO), 처리량(읽기/쓰기 격리), 관측(단계별 trace).

### 2.3 핵심 난점 — 이 프로젝트의 본질

1. **HWP 파싱:** 구조 복잡, 특히 표 복원이 어렵다. 잘못 읽으면 자신 있게 틀린 답. **단일 실패점(SPOF).**
2. **한국어 검색:** 조사 변화("출장비를"/"출장비가")로 단어 검색이 깨짐. 형태소 분석 필수.

> 상세: `design_report_v4.md` §2

## 3. 아키텍처 개요

### 3.1 시스템 컨텍스트

- **외부 경계:** 사용자(인증됨) ↔ API 게이트웨이 ↔ 내부 컴포넌트. 외부 LLM API는 네트워크 경계 밖(유일한 유출 지점).
- **신뢰 경계:** 게이트웨이. 통과 후 모든 컴포넌트는 "권한 해결됨"을 전제.

### 3.2 컴포넌트 구조

```
사용자 → [API 게이트웨이: 인증·ACL·검증]  ← 신뢰 경계
            ├─ 읽기 경로: 질의재작성 → 하이브리드검색(ACL) → 리랭킹 → 생성 → 스트리밍
            └─ 쓰기 경로: 파싱 → 청킹 → 임베딩 → 인덱싱 → 버전전환 → GC  (비동기, 격리)
         공유: GPU 추론(임베딩·리랭커 self-host) + 외부 LLM API
         저장: Qdrant(벡터·ACL) / PostgreSQL(운영메타·버전) / 오브젝트스토리지(원본·IR·청크 보존)
```

읽기/쓰기 분리(NFR-2): 인덱싱이 질의를 막지 않도록 쓰기 경로를 비동기 워커로 격리.

### 3.3 데이터 흐름

- **수집:** 업로드 → 매직바이트 판별 → 포맷별 파싱(IR) → 구조 기반 청킹 → dense+sparse 임베딩 → Qdrant 적재(비활성 버전) → 원자적 전환 → 옛 버전 GC
- **질의:** 인증 → 질의재작성(멀티턴) → 하이브리드 검색(ACL pre-filter) → 리랭킹 → 빈/저점수면 abstain, 아니면 생성 → 인용 검증 → 출처와 함께 응답

### 3.4 데이터 모델

- **IR(중간표현):** 블록 트리 + 블록별 메타. 표는 셀 좌표 보존. 노이즈(머리말) 플래그. (`schemas/ir.py`)
- **청크 메타데이터:** document_id, parent_chunk_id(small-to-big), struct_path, acl_tags, source_block_ids, 버전 필드들. fail-closed 불변식. (`schemas/chunk.py`)
- **계보 체인:** IR block_id → 청크 source_block_ids → 답변 인용. 신뢰성의 척추.

> 상세: `code_design_spec.md` §3·§4

## 4. 핵심 설계 결정 (ADR)

### ADR-001 — VectorDB로 Qdrant 선택

- **맥락:** 하이브리드 검색 + 사용자 ACL + 버전 전환이 필요.
- **결정:** Qdrant.
- **대안:** Firestore(벡터 하이브리드·ACL pre-filter 미지원, GPU 부적합 → 탈락), pgvector(하이브리드·필터 성능 제약), Elasticsearch(가능하나 운영 무게).
- **근거:** dense·sparse가 같은 컬렉션의 named vector → **ACL 필터 불일치(누수 A)가 구조적으로 발생 불가.** 필터를 HNSW 탐색에 통합 → pre-filter recall 붕괴 완화. 실제 엔진으로 검증(7/7).
- **결과:** 분리 엔진 동기화 코드가 불필요해짐. 위협을 막는 게 아니라 위협이 사라지는 선택.

### ADR-002 — 외부 API LLM + self-host 임베딩

- **맥락:** 폐쇄망 해제, 민감 문서 없음 확인.
- **결정:** LLM은 외부 API(인터페이스 뒤), 임베딩·리랭커는 self-host 한국어 특화 기본.
- **대안:** 전부 self-host(GPU 부담), 전부 외부 API(임베딩은 전체 코퍼스 외부 전송 + 한국어 우위 손실).
- **근거:** API 개방의 실익은 LLM에 집중(운영부담 0·한국어 성능). 임베딩은 코퍼스 전체가 외부로 나가고 KURE 등 특화 모델이 우위라 self-host가 안전.
- **결과:** LLM GPU 용량 산정 소멸(rate limit·비용으로 대체). 프로바이더 교체 가능(인터페이스).

### ADR-003 — 버전 태깅 동시성(빈 창 제거)

- **맥락:** 문서 개정 시 "삭제 후 삽입"은 그 사이 빈 창에 검색이 0건 회수(조용한 실패).
- **결정:** 버전 태깅 + 원자적 포인터 전환 + 비동기 GC.
- **대안:** 트랜잭션(엔진 능력 의존, 분리 엔진이면 분산 트랜잭션 불가), INSERT 먼저(중복 창).
- **근거:** 엔진 능력에 베팅하지 않는 애플리케이션 레벨 장치. 하이브리드 두 인덱스 정합성도 단일 전환으로 해결.
- **결과:** 스레드 스트레스에서 순진한 방식 47% 빈 창 → 우리 해법 0.

### ADR-004 — 코드 계약(Pydantic 불변식)

- **맥락:** "권한 없는 청크는 검색 안 됨" 같은 규칙이 문서상 약속이면 안 지켜진다.
- **결정:** 불변식을 Pydantic validator로 박아 **위반 객체 생성 자체를 불가능**하게.
- **대안:** 문서 규칙(미준수), Protobuf(검증을 스키마에 못 박음), JSON Schema(표현력 약함).
- **근거:** 정의·검증·불변식·JSON Schema 자동생성을 한 객체에. fail-closed(acl_tags 빈 값 = 거부).
- **결과:** 계약 70개. 두 번 설계자 버그를 잡음(order_index 전순서, 동시성 자기참조).

### ADR-005 — 게이트웨이를 신뢰 경계로

- **맥락:** ACL은 검색 전에 해결돼야 한다(보고서 §3).
- **결정:** 게이트웨이가 인증·ACL을 단독 해결. `require_auth` 의존성으로 보호 라우트에 강제.
- **대안:** 컴포넌트별 권한 체크(누락 위험·중복), 검색 후 필터링(too late·recall 잠식).
- **근거:** 인증 누락이 구조적으로 불가능. 권한범위 없는 신원 fail-closed 403. 누설 방지(403 대신 404).
- **결과:** 신뢰 경계 동작을 계약 테스트 6개로 고정.

> 상세: `code_design_spec.md`, `api_design_spec.md`

## 5. 컴포넌트 상세 (책임 · 계약 · 위험 · 링크)

| 컴포넌트 | 책임 | 핵심 계약 | 핵심 위험 | 상태 |
|---|---|---|---|---|
| 파서 `parsing/` | 원본 → 구조 보존 IR | 표 셀 좌표 보존, struct_path, 노이즈 분리 | **SPOF — 표 복원율** | 골격(PoC 후) |
| 청커 `chunking/` | IR → 청크 | 구조 기반, contextual, small-to-big | 폴백 발동률 | 미구현 |
| 임베더 `embedding/` | 청크 → dense+sparse | 차원 일치, 형태소 sparse | 한국어 토큰화 | 미구현(GPU 서버) |
| 검색 `retrieval/` | 하이브리드 + ACL | RRF 융합, pre-filter 양쪽 일관 | 필터링 recall | **구현됨** |
| 리랭커 | top-K → top-N | 최저점수 제외 | 꼬리 지연 | 미구현(GPU 서버) |
| 생성 `generation/` | 컨텍스트 → 답변 | abstention, 인용 검증 | unfaithfulness | **구현됨(가드)** |
| 게이트웨이 `api/` | 인증·ACL·라우팅 | require_auth, 누설 방지 | 신뢰 경계 | **구현됨(골격)** |
| 인덱싱 `indexing/` | 버전 전환·GC | 원자적 전환, 멱등 upsert | 빈 창 | **versioning 구현** |

> 컴포넌트별 상세 계약·테스트: `tdd_test_spec.md`

## 6. 보안 설계

- **6.1 신뢰 경계/인증:** 게이트웨이 단독 해결. `require_auth` 강제 주입(누락 불가). 인증 방식은 `IdentityProvider` 추상화(미정, 주입).
- **6.2 사용자 ACL:** 검색 **시점**에 메타데이터 pre-filter. dense·sparse 양쪽 동일 적용(누수 A 차단). 멀티턴 권한 매 턴 재평가. "문서가 외부 가능"과 "사용자가 볼 권한"은 별개 — 후자는 그대로 유지.
- **6.3 데이터 보호/외부 경계:** 외부 LLM이 유일한 유출 지점 → API 키 관리, 나가는 트래픽 감사 로그, 약관·리전 준수(법무).
- **6.4 누설 방지:** 권한 없는 리소스 404(403은 존재 누설), 500 에러는 trace_id만(스택은 로그에만).

> 상세: `api_design_spec.md` §3·§5·§8

## 7. 품질·검증 전략

- **7.1 두 층위 테스트:** **계약 테스트**(불변식, 구현 무관, 항상 초록 — 70개)와 **동작 테스트**(구현을 이끔, TDD 빨강→초록). 섞지 않는다.
- **7.2 품질 게이트:** 평가 하니스로 retrieval(recall@k/MRR)·generation(faithfulness/abstention)을 **유형별 분리** 측정. SLO 미달 시 CI 머지 차단. **abstention은 단독 거부권**(부재 질의에 지어내면 차단).
- **7.3 환각 방어:** 두 모드 분리 — retrieval 실패형(임계값·리랭킹)과 unfaithfulness형(인용 코드 검증). temperature 0은 대책 아님.

> 상세: `tdd_test_spec.md`, `eval/harness.py`

## 8. 운영

- **8.1 배포:** Docker 멀티스테이지(앱/GPU추론 분리), K8s 또는 관리형 컨테이너. 비루트 실행. 외부 LLM은 컨테이너 없음.
- **8.2 확장·용량:** 읽기/쓰기 경로 독립 스케일. LLM은 rate limit·비용 모델(외부 API), 임베딩·리랭커는 GPU 오토스케일. 용량 계산기로 역산(`eval/capacity.py`).
- **8.3 관측:** 요청별 trace_id 전파, 단계별 단일 trace, 검색 점수·abstention율 메트릭. 모든 응답에 청크 ID 로깅(사후 재현).
- **8.4 장애 대응:** 외부 LLM 장애 → graceful degradation(검색 원문+출처 반환). rate limit 백오프·큐잉. 타임아웃·서킷브레이커.

> 상세: `code_design_spec.md` §5·§6

## 9. 리스크 · 가정 · 미정 (정직한 경계)

### 9.1 리스크

- **HWP 파싱 = SPOF (최상위):** 표 복원율이 사업 요구를 못 받치면 프로젝트 전체가 무의미. **국면 A PoC로 가장 먼저 판정.** 완화: PoC 게이트, 파싱 언어 분리 옵션.
- **청킹이 파싱에 곱셈 종속:** 폴백 발동률을 건강 지표로 추적.
- **골드셋 편향:** LLM 생성 질의가 자기 약점 회피 → 적대적·부재 질의 강제 포함.

### 9.2 가정

- 외부 LLM 약관(학습 비사용·리전)이 행정문서에 적법(법무 검토 전제)
- 한국어 임베딩(KURE/BGE-M3)이 본 코퍼스에서 충분한 recall(벤치마크 전제)
- 도메인 담당자가 골드셋 라벨링에 시간 투입 가능

### 9.3 미정 결정 (조기 확정 회피)

파싱 언어 분리(PoC 후), Qdrant self-host vs Cloud, K8s vs 관리형, 구체 모델명(벤치마크 후), 성능 SLO 수치(부하 테스트 후), 인증 방식(조직 IdP 확인 후), rate limiting 위치.

### 9.4 범위 밖 (코드로 못 막음)

- **추론 누수:** 권한 내 청크 조합으로 권한 외 사실 추론 — 청크 단위 ACL로 원천 불가. 제품 정책.
- **기대-역량 격차:** 사용자가 "판단"을 기대 — 제품 면책.
- **LLM-as-judge 채점자 신뢰도:** 사람 표본 교차검증으로 보정.
- **골드셋 대표성:** 도메인 라벨링 투자에 의존.

## 10. 구현 로드맵

리스크 소각 순서 = 빌드 순서:

1. **국면 A — HWP 파싱 PoC** ← 다음 작업. 표 복원율 Go/No-Go. (파싱 언어 분리 결정)
2. 파서 TDD (P-01~P-08 빨강 → 초록)
3. 청커 + 임베더 + Qdrant 인덱싱 (형태소 토크나이저)
4. 검색기 확장 (구현됨, 통합)
5. 생성기 + 외부 LLM 통합 (가드 구현됨, LLMClient 연결)
6. 게이트웨이 파이프라인 주입 (골격 구현됨)
7. 인덱싱 워커 + 버전 전환 + GC
8. 관측·평가·CI 게이트 상시화
9. 컨테이너·배포·부하 테스트 (성능 SLO 실측 동결)

**현재 진행도:** 계약 70개 + 검색기·생성기·게이트웨이 골격 완료. 파서가 다음.

### C. 용어집

- **RAG:** 검색 증강 생성. 문서를 찾아 LLM에 제공해 답하게 함(오픈북 시험).
- **IR:** 중간표현. 파싱 결과의 구조 보존 형식.
- **하이브리드 검색:** 의미(dense) + 단어(sparse) 검색 융합.
- **ACL:** 접근 제어 목록. 누가 무엇을 볼 수 있는가.
- **abstention:** 답을 모를 때 지어내지 않고 "모른다"고 함.
- **계약 테스트 / 동작 테스트:** 불변식 검증 / 구현 동작 검증.
- **SPOF:** 단일 실패점.
- **ADR:** 아키텍처 결정 기록.
