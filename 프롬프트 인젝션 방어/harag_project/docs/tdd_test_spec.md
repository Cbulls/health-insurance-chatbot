# TDD 테스트 명세서 — 한글 행정문서 RAG 챗봇

**목적:** 제품 코드 구현을 이끄는 동작 테스트 설계 (코드 작성 전 빨강 상태로 존재해야 할 테스트)
**전제:** 설계 보고서 v4, 기존 64개 계약 테스트
**원칙:** 테스트는 구현이 아니라 동작을 검증한다. 각 테스트는 요구사항의 번역이며, 보고서의 모든 요구사항은 최소 하나의 실패하는 테스트로 표현된다.

---

## 0. 두 층위 구분 (중요)

| 층위 | 무엇 | 상태 | 이 문서의 대상 |
|---|---|---|---|
| 계약 테스트 (기존 64개) | "데이터가 이렇게 생겨야 / 이 규칙 어기면 거부" — 불변식 | 이미 초록 | 아님 (유지·회귀 방지용) |
| 동작 테스트 (이 문서) | "X를 넣으면 Y가 나와야" — 구현 동작 | **빨강 (구현 전)** | **이 문서가 설계** |

계약 테스트는 구현과 무관하게 영원히 유효하다. 동작 테스트는 아직 구현이 없어 실패하며, 이를 초록으로 만드는 것이 TDD 사이클이다.

TDD 우선순위는 리스크 소각 순서를 따른다 — 가장 불확실하고 치명적인 컴포넌트(파서)부터. 보고서의 "HWP 파싱 PoC를 먼저"가 여기서도 그대로 적용된다.

### 6.1 핵심 동작

| ID | 테스트 | 요구사항 | 빨강 만드는 법 |
|---|---|---|---|
| G-01 | 빈 컨텍스트 → 생성 안 하고 abstention("문서에 없음") | FR-8 | 빈 컨텍스트로 지어내면 실패 |
| G-02 | 저점수 컨텍스트(임계 미달) → abstention | FR-8 | 억지 답변 시 실패 |
| G-03 | 답변의 인용 청크 ID가 실제 컨텍스트에 존재(위조 인용 차단) | FR-8, B-2 경계5 | 컨텍스트에 없는 ID 인용 시 차단돼야 |
| G-04 | 인용 검증 실패 시 답변 차단/재생성 | FR-8 | 검증 우회 시 실패 |
| G-05 | 정상 컨텍스트 → 출처 메타데이터 포함된 답변 | FR-8 | 출처 누락 시 실패 |

### 6.2 환각 두 모드 분리 (품질 게이트)

| ID | 테스트 | 요구사항 | 비고 |
|---|---|---|---|
| G-Q1 | 골드셋 generation: faithfulness ≥ SLO(컨텍스트 근거성) | NFR-3, 국면 C | unfaithfulness형 환각 추적 |
| G-Q2 | 부재 질의(absent) → abstention 정확도 ≥ SLO | 국면 C | **단독 거부권.** 모르는 걸 지어내면 차단 |
| G-Q3 | answer correctness ≥ SLO(유형별) | NFR-3 | 유형별 분리 |
| G-Q4 | 한국어 abstention 거동 별도 검증(영어보다 약함) | NFR-5, 국면 C | 한국어로 "모른다" 잘 하는지 |

G-Q2는 보고서가 가장 강조한 것 — "틀린 답보다 위험한 자신 있는 오답"을 막는 단독 게이트.

### 6.3 품질 측정 방법론 (어떻게 측정하는가)

품질 점수는 골드셋이 작으면 흔들린다. "0.87"이 우연인지 실력인지 구분하는 방법론. 구현: `eval/quality.py`.

**신뢰구간 기반 게이트 (핵심)**
- 단일 점수가 아니라 **부트스트랩 신뢰구간** [하한, 상한]을 보고한다. 골드셋을 복원추출로 재샘플링해 점수 분포를 만든다.
- **통과 판정 = 신뢰구간 하한이 SLO를 넘을 때**(점추정이 아니라). 보수적 — 작은 표본의 우연한 통과를 막는다. 검증됨: 같은 점수 0.8이라도 n=10이면 CI[0.50, 1.00]로 SLO 0.75 하한 미달(차단), n=100이면 [0.72, 0.88]로 좁아짐.
- 함의: 골드셋이 작으면 게이트를 못 넘는다 → **골드셋을 충분히 키우라는 압력**으로 작동(self-critique 3의 대표성 문제를 게이트가 강제).

**retrieval 채점 (결정적, LLM 불필요)**
- recall@k, MRR을 질의별 점수 리스트로 계산(`recall_at_k_scores`, `reciprocal_rank_scores`).
- 부재 질의(gold 빈 집합)는 retrieval 채점에서 제외.
- 결정적이라 빠르고 재현 가능 → CI의 1차 게이트.

**generation 채점 (LLM-as-judge, 주의)**
- faithfulness/correctness는 의미 판단이라 LLM 심판이 채점. **단, 심판 자체가 편향될 수 있다**(§12 경계).
- 완화: ① 심판 프롬프트를 고정·버전관리, ② 사람이 채점한 표본(예: 20%)과 심판 점수의 일치도(Cohen's κ)를 주기적으로 측정해 심판 신뢰도를 검증, ③ κ가 임계 미달이면 심판 점수를 신뢰하지 않음.
- abstention 정확도는 결정적(답 유무로 판정) → LLM 심판 불필요, 단독 게이트.

**골드셋 구축 절차 (측정의 토대)**
- 유형별 층화: body/table/multi_clause/absent/conflicting. 유형별 최소 표본 수 확보.
- 질의 출처 3갈래 강제: 실사용 로그 + LLM 생성 + **적대적 수작업**(자기 회피 편향 차단).
- 라벨러 간 일치도(IAA): 표본 일부를 2인 독립 채점, 일치도 낮은 질의는 정답 모호 → 제외/재정의.
- 버전·만료: 문서 개정 시 연결된 골드 질의를 stale 표시(국면 C 거버넌스).

**측정 환경 고정**
- 평가는 고정된 인덱스 스냅샷에서. 인덱스가 변하는 중 측정하면 점수가 흔들린다.
- 임베딩·청킹·프롬프트 버전을 결과에 기록(어떤 설정의 점수인지 추적, A/B 비교 가능).

| EE-02 | 질의 → 검색→리랭킹→생성→출처 포함 답변 반환 | 전체 | — |
| EE-03 | 계보 체인: 답변 인용 → source_block_ids → IR 블록 → 원본 페이지 역추적 가능 | B-1, FR-8 | 체인 끊기면 실패 |

EE-03은 self-critique 1번("정확한 출처를 단 틀린 답")을 사후 추적 가능하게 하는 신뢰성의 척추.

### 8.4 성능 측정 방법론 (어떻게 측정하는가)

"p95 ≤ SLO"는 목표지 방법이 아니다. 측정을 재현 가능하게 만드는 절차를 못 박는다. 구현: `eval/perf.py`.

**측정 구간 정의**
- **워밍업 제외:** 첫 N개 요청(JIT·캐시·커넥션풀 미초기화)은 측정에서 버린다. 워밍업 미제외 시 p99가 거짓으로 튄다(검증됨: 3284ms → 1195ms). 권장 워밍업: 정상 지연이 안정될 때까지(보통 수십 요청).
- **정상 상태(steady state):** 워밍업 후 일정 시간/요청 수를 측정 구간으로. 부하 종료 직전 cooldown도 제외.

**부하 프로파일**
- **closed-loop vs open-loop:** 동시 사용자 N명 고정(closed)이 아니라, 목표 RPS를 주입(open)해야 실제 트래픽을 반영한다. closed-loop는 시스템이 느려지면 부하도 같이 줄어 병목을 숨긴다.
- **램프업:** 0 → 목표 부하를 점진 증가시켜 한계점(p95가 SLO를 깨는 지점)을 찾는다.

**지표 계산**
- **평균 금지, 백분위수:** p50/p95/p99. 평균은 꼬리 지연을 숨긴다.
- **단계별 분해:** end-to-end만이 아니라 임베딩·검색·리랭킹·TTFT를 각각 측정. 병목 식별(검증: rerank가 병목으로 드러남). `LatencySample.stages`로 수집.
- **TTFT 분리:** 스트리밍은 첫 토큰 시각을 별도 측정(체감 지연).

**판정**
- p95가 SLO 이하 + p99가 별도 상한 이하. 둘 다 통과해야 함.
- 부하별 곡선을 그려 "몇 RPS까지 SLO 유지"를 용량으로 보고(국면 D 계산기 검증).

**도구**
- 부하 주입: locust/k6류(open-loop, 램프업 지원).
- 측정/분석: `eval/perf.py`의 `analyze(samples, warmup=N)`.

### 8.5 처리량·확장성 측정 방법론

- **PERF-06(읽기/쓰기 격리):** 인덱싱 워커에 대량 문서를 주입하면서 **동시에** 질의 부하를 걸고, 질의 p95가 인덱싱 없을 때 대비 열화율(예: <10%)을 측정. 격리가 깨지면 질의 p95가 급등.
- **PERF-08(규모별 recall):** 문서 수를 10배씩 늘리며 골드셋 recall@k를 측정, HNSW `ef`를 올려 보정. recall↔지연 곡선에서 SLO 만족 점을 찾음(국면 D ef_curve).
- **PERF-07(rate limit):** 외부 LLM rate limit을 모의(429 주입)하고 백오프·큐잉이 크래시 없이 처리하는지. 큐 깊이·대기 시간 측정.

## 12. 테스트가 아닌 것 (정직한 경계)

다음은 자동화 테스트로 닫지 않는다(보고서 §8과 일치):

- **LLM-as-judge 채점자 신뢰도:** faithfulness/correctness 채점을 LLM이 하면 그 채점자 자체의 편향은 별도 검증 영역. 사람 표본 교차검증으로 보정.
- **골드셋 대표성:** 테스트는 "유형별로 본다"를 강제하나 각 유형의 충분성은 도메인 라벨링 투자에 의존.
- **외부 API 약관 준수:** 학습 비사용·데이터 주권은 계약·법무 검토. 코드 테스트 불가.
- **추론 누수:** 청크 단위 ACL로 원천 불가. 제품 정책.
- **기대-역량 격차:** 사용자가 "판단"을 기대하는 것은 제품 범위·면책으로.
