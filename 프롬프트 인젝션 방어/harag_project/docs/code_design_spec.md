# 한글 행정문서 RAG 챗봇 — 코드 설계서 (구현 아키텍처)

**범위:** 구현에 사용할 언어·프레임워크·인프라·배포 전략과 그 선택 근거
**전제:** 설계 보고서 v4(클라우드, 외부 API LLM 허용, 민감 문서 없음), 기존 코드 계약·검증 자산
**문서 성격:** 설계 보고서가 "무엇을 왜"라면, 이 문서는 "무엇으로 어떻게". 각 선택에 근거·트레이드오프·탈락 대안을 명시한다.

---

## 0. 선택의 제약 조건 (무엇이 기술 선택을 이미 좁혔는가)

기술 스택은 백지에서 고르는 게 아니다. 앞선 설계가 이미 많은 것을 결정했다.

- **Pydantic 기반 계약이 존재한다** → Python 생태계가 사실상 확정. 스키마·경계·검증 64개가 Python으로 작성됐고, 이를 버리고 다른 언어로 가는 비용은 정당화 불가.
- **Qdrant를 VectorDB로 확정** → 검색 계층은 qdrant-client(Python) 위에 선다.
- **임베딩·리랭커는 self-host 한국어 특화 기본** → GPU 추론 서빙이 필요(파이프라인 일부). LLM은 외부 API.
- **HWP 파싱이 SPOF** → 파싱 라이브러리 생태계가 빈약한 영역이라, 가용 라이브러리가 언어 선택을 제약(Python/JVM 계열에 그나마 존재).
- **평가·관측이 1급 요구사항** → LLM 전용 트레이싱·평가 도구와 통합 쉬운 스택 선호.

결론: **Python이 자연스러운 중심**이다. 이는 취향이 아니라 기존 자산과 생태계의 귀결이다.

## 1. 언어 및 런타임

### 1.1 결정: Python 3.12+

| 근거 | 설명 |
|---|---|
| 기존 자산 | Pydantic 계약 64개가 Python. 재작성 비용 회피 |
| 생태계 | RAG 스택(임베딩·리랭커·LLM SDK·평가)의 1차 시민이 Python |
| HWP/문서 파싱 | 가용 파서가 Python/JVM에 집중 |
| 타입 안전 | 3.12의 타입 힌트 + Pydantic v2로 런타임 검증까지 |

### 1.2 탈락 대안

- **Node/TypeScript:** 프런트엔드엔 강하나 ML/파싱 생태계가 약함. 임베딩·리랭커 self-host에 부적합.
- **Go/Rust:** 성능·동시성 우수하나 ML 생태계 부재. 고성능이 필요한 특정 컴포넌트(예: 토크나이저)만 부분 채용 여지.
- **JVM(Kotlin/Java):** HWP 파서가 일부 존재하나, 나머지 RAG 생태계가 Python에 밀림. 파싱만 JVM 서비스로 분리하는 하이브리드는 가능(아래 1.3).

### 1.3 예외: 파싱 계층의 언어 분리 가능성

HWP 파싱은 Python 라이브러리(pyhwp/hwp5 등)의 표 복원 품질이 불확실하다(국면 A PoC가 판정). PoC 결과 Python 파서가 임계 미달이고 JVM/네이티브 파서가 더 낫다면, **파싱만 별도 서비스(다른 언어)로 분리**하고 IR(JSON)로 통신한다. B-1의 IR 스키마가 이미 언어 중립 JSON Schema라, 이 분리가 깨끗하게 된다. **이 결정은 PoC 이후로 미룬다** — 지금 확정하지 않는다.

## 2. 애플리케이션 프레임워크

### 2.1 결정: FastAPI

| 근거 | 설명 |
|---|---|
| Pydantic 네이티브 | 요청·응답 검증에 기존 스키마를 그대로 재사용. 별도 직렬화 계층 불필요 |
| 비동기 | 외부 LLM API 호출·검색이 I/O 바운드 → async로 동시성 효율 |
| 스트리밍 | SSE 지원 → FR-8 TTFT 스트리밍(NFR-1 체감 지연) 자연스럽게 |
| 자동 문서화 | OpenAPI 자동 생성 → 게이트웨이·클라이언트 계약 |

### 2.2 탈락 대안

- **Flask:** 동기 중심, 비동기·스트리밍이 부자연스러움. LLM I/O 바운드 워크로드에 비효율.
- **Django:** ORM·관리자 등 무거운 기능이 불필요. RAG API에 과함.
- **LiteLLM/LangServe 같은 풀스택 RAG 프레임워크:** 빠른 시작엔 좋으나, 우리는 이미 컴포넌트 계약을 직접 설계했다. 프레임워크의 추상화가 우리 계약(경계 검증·ACL·버전 전환)과 충돌하면 오히려 부채. **명시적 제어를 택한다.**

### 2.3 RAG 오케스트레이션: 최소 채택 원칙

LangChain/LlamaIndex 같은 프레임워크는 **선택적·부분적으로만** 쓴다. 이유: 우리 파이프라인의 핵심 로직(구조 기반 청킹, 하이브리드+ACL 검색, 인용 검증 생성)은 이미 계약으로 직접 구현했다. 프레임워크의 범용 추상화를 전면 도입하면 그 계약을 우회하게 된다. 단, 잘 검증된 부품(예: 특정 문서 로더, 토크나이저 래퍼)은 선별 채용. **프레임워크는 도구지 아키텍처가 아니다.**

## 3. 컴포넌트별 기술 매핑

| 컴포넌트 | 기술 | 근거 |
|---|---|---|
| API 게이트웨이 | FastAPI + 인증 미들웨어 | ACL 해결 지점(B-4), Pydantic 검증 |
| 파서 | Python 파서(PoC 후 확정) + 포맷 라우터 | FR-2, 국면 A 의존 |
| 청커 | 자체 구현(구조 기반) + 형태소 토크나이저 | FR-3, 프레임워크 청커는 토큰 고정이라 부적합 |
| 임베더 | self-host 한국어 임베딩(KURE/BGE-M3 계열) + GPU 서빙 | FR-5, 한국어 우위 |
| 리랭커 | self-host cross-encoder + GPU 서빙 | FR-7 |
| 검색기 | qdrant-client 하이브리드(기존 retriever.py) | 확정 |
| 생성기 | 외부 LLM API(LLMClient 인터페이스 뒤, 기존 generator.py) | v4, 안전 가드 완성됨 |
| 형태소 분석 | Kiwi(또는 Mecab) | FR-5 한국어 BM25 전제 |

GPU 추론 서빙(임베딩·리랭커)은 vLLM/TEI(Text Embeddings Inference)류 서버 위에 두고, 애플리케이션은 HTTP로 호출 → 추론과 앱 로직 분리(NFR-2 격리, 독립 스케일).

## 4. 데이터 저장소

### 4.1 결정: 세 종류 분리

| 저장소 | 기술 | 용도 |
|---|---|---|
| 벡터 + 메타데이터 | **Qdrant** | 청크 임베딩·ACL·버전 태그. 검색의 중심 |
| 원본·IR·청크 보존 | **오브젝트 스토리지(S3 호환)** | 재인덱싱 전제(NFR-8). 원본·IR·청크를 보존 |
| 운영 메타데이터 | **PostgreSQL** | 문서 등록부, 버전 이력, 평가 골드셋, 감사 로그 |

### 4.2 왜 셋으로 나누나

핵심은 **재인덱싱 가능성(NFR-8)**이다. 임베딩 모델은 반드시 바뀌고, 그때 원본→IR→청크를 재실행해야 한다. 임베딩(Qdrant)만 들고 있으면 전략 변경 시 복구 불가다. 그래서 원본·IR·청크를 오브젝트 스토리지에 **별도 보존**한다. PostgreSQL은 "어떤 문서가 어떤 버전으로 인덱싱됐나"의 진실 원천(source of truth)으로, B-3 버전 전환의 활성 버전 레지스트리 역할도 겸할 수 있다.

### 4.3 Firebase 평가 (질문에서 제기됨) — 결론: 부적합

Firebase(Firestore + Auth + Functions)를 검토했으나, 이 프로젝트에는 맞지 않는다. 정직한 근거:

- **벡터 검색:** Firestore는 하이브리드(dense+sparse)+RRF+ACL pre-filter를 지원하지 않는다. Qdrant가 핵심인데 Firebase가 그 자리를 대체 못 한다.
- **GPU 추론:** self-host 임베딩·리랭커는 GPU가 필요하다. Firebase Functions는 GPU 워크로드에 부적합.
- **벤더 락인:** Firestore의 데이터 모델·쿼리에 묶이면, 우리 스키마 계약(B-1)과 버전 전환(B-3)을 그 위에 얹기 어렵다.
- **적합한 곳이 없지 않다:** 만약 **간단한 사용자 인증**만 필요하면 Firebase Auth는 후보가 될 수 있다. 하지만 행정문서 ACL은 부서·권한 기반이라 자체 인증/인가(또는 기업 SSO·OIDC) 통합이 더 맞다.

요약: Firebase는 모바일·웹앱 백엔드엔 훌륭하나, **GPU 추론 + 전문 벡터검색 + 데이터 주권**이 핵심인 이 RAG에는 구조적으로 안 맞는다. 도구를 요구사항에 맞추는 것이지, 요구사항을 도구에 맞추지 않는다.

## 5. 컨테이너화 및 배포

### 5.1 결정: Docker + Kubernetes (또는 관리형 컨테이너)

**Docker는 사용한다.** 근거:

- **재현성:** 형태소 분석기·파서·GPU 드라이버 등 환경 의존성이 많아, 컨테이너로 고정하지 않으면 "내 PC에선 됐는데" 문제가 심각.
- **컴포넌트 격리(NFR-2):** 읽기/쓰기 경로, GPU 추론 서버, API를 별도 컨테이너로 → 독립 배포·스케일.
- **GPU 스케줄링:** 임베딩·리랭커 GPU 워크로드를 K8s가 노드 풀로 관리, 오토스케일.

### 5.2 컨테이너 구성(논리)

| 컨테이너 | 역할 | 스케일 특성 |
|---|---|---|
| api-gateway | FastAPI, 인증·ACL·라우팅 | 수평 스케일(stateless) |
| query-service | 검색·리랭킹·생성 오케스트레이션(읽기) | 수평 스케일 |
| indexing-worker | 파싱·청킹·임베딩·버전전환·GC(쓰기) | 큐 기반, 독립 스케일 |
| embedding-server | GPU, self-host 임베딩(TEI류) | GPU 노드 풀, 오토스케일 |
| reranker-server | GPU, cross-encoder | GPU 노드 풀 |
| (parser-service) | PoC 후 언어 분리 시에만 | 선택적 |

Qdrant·PostgreSQL·오브젝트 스토리지는 관리형 서비스 또는 별도 스테이트풀 셋. 외부 LLM은 API라 컨테이너 없음(네트워크 경계 밖).

### 5.3 배포 환경

- **클라우드:** v4에서 폐쇄망 해제 → 관리형 K8s(EKS/GKE/AKS) 또는 관리형 컨테이너(Cloud Run류) + 관리형 Qdrant Cloud 옵션.
- **데이터 주권:** 행정문서 → 리전은 국내로 제약(보고서 §3.3, 법무 검토 전제). 외부 LLM API도 국내 리전·기업 약관 확인.
- **읽기/쓰기 분리:** indexing-worker를 메시지 큐(예: 클라우드 큐 서비스) 뒤에 두어 질의 경로와 격리.

### 5.4 탈락/보류 대안

- **서버리스 함수(Lambda/Functions)만으로:** GPU 추론·긴 인덱싱 작업·상태 있는 검색에 부적합. 일부 이벤트 트리거(업로드 감지)엔 부분 활용 가능.
- **단일 모놀리식 컨테이너:** 읽기/쓰기·GPU 격리(NFR-2)가 불가. 초기 프로토타입엔 허용하되 운영엔 분리.

## 6. 외부 통합 및 관측

### 6.1 외부 LLM

LLMClient 인터페이스(기존 generator.py) 뒤에 외부 API를 끼운다. 구현 시: rate limit 백오프·큐잉, 토큰 비용 상한, 타임아웃·서킷브레이커(NFR-4), 나가는 트래픽 감사 로그(v4 §6.2). 모델·프로바이더는 교체 가능하게 인터페이스로 추상화 — 특정 API에 락인 안 됨.

### 6.2 관측 스택

| 관심사 | 도구(범주) | 요구사항 |
|---|---|---|
| LLM 트레이싱 | Langfuse/LangSmith류 | NFR-6 단계별 trace |
| 메트릭 | Prometheus + Grafana | 지연·abstention율·점수 분포 |
| 로그 | 구조화 로깅 + 중앙 수집 | 청크 ID·점수 사후 재현(OBS-01) |
| 평가 | 기존 eval_harness.py + CI | 골드셋 회귀 게이트 |

### 6.3 CI/CD

- **CI 게이트:** 기존 계약 테스트(64개) + 동작 테스트 + 품질 게이트(P-Q/R-Q/G-Q). 품질 SLO 미달 시 머지 차단(NFR-3).
- **컨테이너 빌드:** 멀티스테이지 Dockerfile, 이미지 스캔.
- **배포:** 스테이징 → 평가셋 재통과 → 프로덕션. 임베딩 모델 교체 시 재인덱싱 + 평가 게이트(보고서 §6.2).

## 8. 구현 순서 (리스크 소각 = 빌드 순서)

설계 보고서·TDD 명세의 우선순위를 그대로 따른다.

1. **파서 + PoC** — 국면 A 게이트. 표 복원율로 Go/No-Go. 여기서 파싱 언어 분리(1.3) 결정.
2. **청커 + 임베더 + Qdrant 인덱싱** — 쓰기 경로. 형태소 토크나이저 통합.
3. **검색기** — 기존 retriever.py 확장(하이브리드·ACL 완성됨).
4. **생성기 + 외부 LLM 통합** — 기존 generator.py에 실제 LLMClient.
5. **API 게이트웨이 + 인증·ACL** — 읽기 경로 완성.
6. **인덱싱 워커 + 버전 전환 + GC** — 쓰기 경로 완성(B-3).
7. **관측·평가·CI 게이트** — 회귀 방지 상시화.
8. **컨테이너화·배포·부하 테스트** — 성능 SLO 실측 동결(국면 D 계산기 기준선).

각 단계는 TDD 사이클(빨강→초록→리팩터)로, 기존 동작 테스트 명세(tdd_test_spec.md)를 초록으로 만들며 진행.

## 9. 의도적으로 미루는 결정 (정직한 경계)

지금 확정하지 않는다. 정보가 부족한 상태의 조기 확정은 부채다.

- **파싱 언어 분리 여부:** 국면 A PoC 결과에 의존(1.3).
- **Qdrant self-host vs Qdrant Cloud:** 데이터 주권·운영 역량·비용 비교 후. 기능은 동일.
- **K8s vs 관리형 컨테이너(Cloud Run류):** 팀 운영 역량·GPU 노드 요구에 따라. GPU 워크로드 유무가 분기점.
- **구체 LLM·임베딩 모델명:** 본인 코퍼스 벤치마크 후(보고서 §8과 일치).
- **성능 SLO 수치:** 부하 테스트 실측 후 사업 합의로 동결(국면 D 계산기로 기준선만 역산).
- **인증 방식(자체 vs 기업 SSO/OIDC):** 조직 IdP 환경 확인 후.

## 10. 결론

이 스택의 중심은 **Python + FastAPI + Qdrant + Docker/K8s + self-host GPU 추론 + 외부 LLM API**다. 모든 선택의 일관된 원리는 두 가지다.

첫째, **기존 자산과 검증된 결정을 존중한다** — Pydantic 계약·Qdrant·계약 테스트가 언어와 핵심 스택을 이미 정했고, 이를 뒤집는 비용은 정당화되지 않는다.

둘째, **도구를 요구사항에 맞춘다** — Firebase가 좋은 도구여도 GPU 추론·전문 벡터검색·데이터 주권이 핵심인 이 RAG에는 안 맞으므로 채택하지 않는다. 요구사항을 도구에 맞추는 역방향은 하지 않는다.

그리고 변하지 않는 것: 이 모든 스택 위에서도 **국면 A의 HWP 파싱 PoC가 첫 작업**이다. 어떤 멋진 인프라도 HWP가 안 읽히면 무의미하다.
